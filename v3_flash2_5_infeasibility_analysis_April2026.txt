SUMMARY OF FAILURE MODE
------------------------
All 4 runs exhibited complete and persistent design infeasibility — not a single
feasible design was generated across their entire execution (4378, 703, 642, 597
evaluations respectively). Every evaluation returned reward = -10.0 (capped penalty).

This is a "cold-start failure" that never recovers: once the first 6 LLM-generated
designs (initialize_n_sample=6) all land in infeasible territory, the power-law
parent selection has no positive signal to work with. Every subsequent design is
generated from an all-infeasible database, giving the LLM no gradient toward
the feasible region.

CONSTRAINT VIOLATIONS (cold-start iterations 0-5)
--------------------------------------------------
Ranked by frequency across all 4 failed runs:

1. cl_target_unreachable (~99.8% of all designs)
   - LLM consistently proposed airfoils with insufficient camber to reach CL
     targets (0.8-1.6 range)
   - PRIMARY BLOCKER: without sufficient lift, the design cannot be feasible
   - Upper weights [0] too small (0.1-0.4 range; high-lift airfoils need ~0.3-0.6)

2. wiggliness (65-99% of designs)
   - Non-smooth Kulfan weight sequences with abrupt transitions between adjacent
     weights, causing wavy surfaces
   - LLM did not self-constrain smoothness of weight progressions

3. analysis_confidence (~91%)
   - Proposed geometries were outside NeuralFoil's training distribution
   - Surrogate could not evaluate them reliably
   - Indicates extreme or invalid shapes being proposed

4. te_angle (~90%) and t090 (~87%)
   - Trailing edge angle and thickness-at-90%-chord violations
   - Coupled to the camber problem: attempts to increase lift worsen these

IS IT WORTH RUNNING MORE CASES?
---------------------------------
Probably not at the current (unmodified) config. Reasons:

  1. Failure rate already well-estimated: 1/18 (~5.6%) with RETRY evidence
     supporting the lower bound. More runs narrow the CI but don't change
     the strategic picture.
  2. Failure mechanism fully understood: systematic LLM cold-start bias
     (99.8% cl_target_unreachable), not random noise. More runs characterize
     frequency but teach nothing new about cause.
  3. Best result (-0.0790) doesn't beat the adjoint (-0.0787). Additional
     runs are unlikely to break through this ceiling without a config change.
  4. Higher-value path: implement Fix 3 or Fix 6, re-characterize failure
     rate under the improved config. Running more cases at a config you
     intend to improve has limited value.

  Exception: if a larger n is needed for a statistical comparison in a paper,
  additional runs could be justified — but ~18 runs with well-understood
  failure classification is defensible for most purposes.

  before the ~2400-eval window where slow starters can find feasibility)

POTENTIAL MITIGATIONS — FRAMEWORK-LEVEL (April 2026 brainstorm)
-----------------------------------------------------------------
The following are general fixes for the v3_dynamic_optimizer framework, not
specific to NeuralFoil or reward_exact_notebook. Ordered roughly by
implementation effort. Cost/impact assessed relative to a 6000-eval budget.

FIX 1: Auto-restart on stuck runs  [Effort: Low | Impact: High]
  If no feasible design is found by eval N (tunable --feasibility_timeout_evals),
  automatically cancel and relaunch with a new random seed. Purely orchestration-
  level — no framework code changes needed. Most practical near-term fix.
  Eliminates wasted compute on runs that will never recover.

FIX 2: Increase initialize_n_sample  [Effort: Trivial | Impact: Medium]
  More context-free cold-start iterations give more independent chances to land
  in the feasible region before power-law selection locks onto infeasible parents.
  Currently 6 (180 evals); could try 15–30 (450–900 evals).

  Cost scaling: linear — each extra cold-start iteration = 1 batch (batch_size
  evals + 1 LLM call). Out of 6000 total evals, going from 6 to 15 cold-start
  iterations adds only ~270 evals (~4.5% of budget). However, if the LLM has a
  systematic bias toward infeasible designs (as suggested by 99.8% cl_target_
  unreachable rate), more independent cold-start attempts may not help much.
  Fixes #3, #4, #6 are likely more impactful per unit cost.

FIX 3: Pass uncapped penalty magnitude to LLM context  [Effort: Low-Med | Impact: High]
  The reward cap flattens all infeasible designs to the same value (-10), so the
  LLM sees no gradient — "completely infeasible" and "almost feasible" look
  identical. Passing fitness_total (uncapped) as context lets the LLM identify
  which parent was *least* infeasible and attempt to improve from it. This is the
  gradient_infeasible idea already implemented for L-BFGS-B, applied here to the
  LLM's context window.

FIX 4: Per-constraint feedback in LLM prompt  [Effort: Medium | Impact: High]
  Rather than just showing the scalar reward, explicitly tell the LLM which
  constraints were violated and by how much for each parent/inspiration design.
  The reflection agent partially does this, but it could be made more prominent
  in the main designer prompt. Fully general — requires constraint_violations
  dict to be surfaced in context, which is already in results.json for any
  environment that supports it.

FIX 5: Feasibility-aware island management  [Effort: Medium | Impact: Med-High]
  Two targeted changes to the island/migration logic:
  a) Migration filter: only migrate feasible designs. If no feasible design
     exists, skip migration entirely (don't spread infeasible solutions).
  b) Island reset: if one island has zero feasibility after N iterations, reset
     it with fresh context-free LLM proposals rather than parent-based ones.
  Currently all 3 islands can simultaneously converge to infeasible regions and
  cross-pollinate each other with bad designs via migration.

FIX 6: Adaptive feasibility-mode prompting  [Effort: Medium | Impact: High]
  Add a "feasibility mode" strategy that activates when the database has zero
  feasible designs. In this mode, the LLM is prompted to focus only on satisfying
  constraints (not optimizing the objective). Once a feasible design is found,
  switch back to normal objective-optimization mode. Environment-agnostic — only
  requires adding a strategy branch based on feasibility count in the database.

FIX 7: Feasibility-first seeding from a baseline  [Effort: Low | Impact: High]
  Allow an optional --baseline design (a known feasible point) to be injected
  into the database at iteration 0 before any LLM proposals. The framework
  already has a --baseline argument but it may not be wired to seed the database.
  Guarantees the power-law sampler always has a feasible parent, eliminating the
  cold-start failure mode — at the cost of reduced exploration diversity from
  a potentially narrow starting region.

FIX 8: Progressive constraint tightening  [Effort: High | Impact: Medium]
  Start with relaxed constraint penalty weights and tighten them over iterations
  as the optimizer finds feasible regions. Requires constraint weights to be
  schedulable over iterations — a more significant framework change but fully
  general. Analogous to continuation methods in numerical optimization.

FIX 9: Hybrid LLM + gradient feasibility repair  [Effort: High | Impact: High]
  After generating an infeasible design, run a short L-BFGS-B feasibility-seeking
  step (minimizing penalty only, no objective) to "repair" it before adding to
  the database. Combines the two methods' strengths: L-BFGS-B is effective at
  local constraint satisfaction once it has a gradient signal; the LLM provides
  global exploration.

HOW TO CONFIRM A RUN HAS FAILED (REVISED KILL CRITERION)
---------------------------------------------------------
A flash2_5 run can only be confidently killed when ALL THREE conditions hold:

  1. Eval count > 2400 — the maximum first-feasibility window across all 14
     successful runs (attempt23: eval 2348). Before this point, even a run that
     looks stuck may eventually find feasibility.

  2. No improvement in rolling-best fitness_total over a long window — attempt8's
     best fitness_total was frozen at -2.50 from eval ~1000 to 4378 (3378-eval
     plateau). Successful slow starters (attempt23) showed gradual drift toward
     zero even before finding feasibility.

  3. Boundary-hitting behavior (qualitative) — confirmed failure showed LE weight
     pinned to 0.5 and TE → 0, indicating the LLM was thrashing at parameter
     bounds rather than exploring the interior of the design space.

Composite kill criterion: no feasible design AND no improvement in fitness_total
rolling-best over 1000+ evals AND eval > 2400.

IS INCREASING ROBUSTNESS THE BETTER PATH?
------------------------------------------
Yes — robustness improvements are preferable to simply running more cases.

The failure is systematic, not random: the 99.8% cl_target_unreachable rate
across all failed runs reflects a structural LLM bias toward low-camber airfoils
that can't reach the CL targets (0.8–1.6). This is a cold-start problem where
the LLM receives no positive signal and iterates in the wrong region indefinitely.
More runs at the same configuration repeat this failure mode with ~6% probability
per run, rather than fixing the underlying issue.

Priority fixes for robustness (from the 9-fix brainstorm above):

  Attack the cold-start directly (highest impact, fix root cause):
    Fix 7 (Baseline seeding): inject a known feasible design at iter 0 —
      eliminates cold-start failure by construction
    Fix 6 (Feasibility-mode prompting): switch to constraint-satisfaction-only
      mode when no feasible designs exist yet

  Improve the gradient signal to LLM (medium effort, high impact):
    Fix 3 (Uncapped penalty in LLM context): pass fitness_total instead of
      capped reward — gives the LLM a real gradient to work with
    Fix 4 (Per-constraint feedback): tell the LLM which constraints were
      violated and by how much (currently partial)

  Recommended path (lowest effort / highest combined impact):
    Fix 3 + Fix 6 together address both root causes (no signal + no targeted
    constraint-satisfying behavior). Fix 7 additionally eliminates cold-start
    entirely. Fix 1 (auto-restart) is a cheap safety net regardless.

  At the current ~5.6% confirmed failure rate with 200-iteration budgets:
    - Running many more baseline cases is low value — the number is likely to
      stay near 5-10% without a fix
    - Implementing Fix 3 or Fix 6 and re-characterizing is the better use of
      compute: could drive failure rate to near zero and simultaneously improve
      the quality of successful runs
