Bayesian Optimization (GP + EI) Implementation Plan for NeuralFoil
====================================================================
Date: March 24, 2026


OVERVIEW
--------
BO-GP fits into the framework as a new frameworks/BO/ module, following the
same pattern as frameworks/GA/ (PSO). The key interface points are:
  - environment.get_param_bounds()          -> lb, ub numpy arrays (18-dim)
  - environment.write_design(x, dir, name)  -> writes JSON, returns path
  - environment.simulate(design_path, dir)  -> returns (reward, results)
  - environment.get_results_csv_columns() / get_results_csv_row(metrics)


LIBRARY CHOICE: scikit-optimize vs BoTorch
------------------------------------------
                   scikit-optimize (skopt)     BoTorch
                   -----------------------     -------
Ease               Simple, sklearn-style API   More complex, PyTorch-based
Maturity           Stable, widely used         Actively developed by Meta
18-dim perf        Adequate                    Better (better acq. optimizer)
Dependencies       scipy + sklearn only        PyTorch + gpytorch
Recommended for    Quick implementation        More serious research use

Recommendation: Start with scikit-optimize — it is ~30 lines of loop code.
If BoTorch is needed for the paper, migrate later. skopt is already
well-cited in the literature.

IMPLEMENTATION STEPS
--------------------

Step 1 — Install
  pip install scikit-optimize
  # or for botorch:
  pip install botorch gpytorch

Step 2 — Create frameworks/BO/ structure
  frameworks/BO/
      __init__.py      (empty)
      run.py           (the framework entry point)

Step 3 — Write run.py
  The core structure mirrors frameworks/GA/run.py. The key difference is
  that instead of a PSO swarm, a skopt.Optimizer is maintained that
  suggests the next point based on all previous observations.

  Pseudocode outline:

    from skopt import Optimizer

    def add_args(parser):
        parser.add_argument('--n_initial', type=int, default=30,
                            help='Random initial samples before GP kicks in')
        parser.add_argument('--n_calls', type=int, default=500,
                            help='Total evaluations (initial + BO)')
        parser.add_argument('--acq_func', type=str, default='EI',
                            choices=['EI', 'PI', 'LCB'])

    def run(environment, args, output_dir):
        lb, ub = environment.get_param_bounds()
        dimensions = list(zip(lb.tolist(), ub.tolist()))  # skopt format

        opt = Optimizer(
            dimensions=dimensions,
            base_estimator='GP',
            acq_func=args.acq_func,
            n_initial_points=args.n_initial,
            random_state=42,
        )

        # Evaluation loop:
        # 1. x = opt.ask()                              -> next suggested point
        # 2. environment.write_design(x, case_dir, ...) -> design_path
        # 3. reward, results = environment.simulate(...)
        # 4. opt.tell(x, -reward)                       -> skopt minimizes, negate reward
        # 5. log to CSV

Step 4 — Wire into run_benchmark.py
  Nothing to change. run_benchmark.py auto-discovers frameworks by scanning
  frameworks/*/. As long as frameworks/BO/__init__.py and
  frameworks/BO/run.py exist with add_args and run defined, the framework
  appears automatically.

Step 5 — Run it
  python run_benchmark.py \
    --framework BO \
    --environment NeuralFoil \
    --reward reward_exact_notebook \
    --n_initial 30 \
    --n_calls 500 \
    --model_size large

Step 6 — Plotting
  animate_neuralfoil_comparison_eval_axis.py reads results.csv with columns
  [iteration, reward, best_reward, ...]. The BO run.py just needs to write
  the same CSV format. BO produces 1 eval per iteration, which maps cleanly
  onto the eval-axis comparison plot.


KEY DESIGN DECISIONS
--------------------
1. n_initial
   30 is reasonable for 18 dims (roughly 1-2x n_dims). Too few and the GP
   fits noise; too many and it degenerates into random search.

2. Kernel
   skopt defaults to Matern 5/2, which is standard. No change needed.

3. Normalization
   skopt handles input normalization internally. Consider normalizing the
   reward if it has a wide range.

4. GP scaling at 500+ evals
   GP training is O(n^3) — at ~500 points it starts getting slow (seconds
   per iteration). BoTorch handles this better via approximate GPs. For the
   paper, this scaling cost is an interesting result to report, as it
   highlights a structural weakness of GP-BO vs LLM-guided methods at
   higher evaluation budgets.

5. Random seeds
   Run 3-5 seeds and report mean +/- std to match how other baselines are
   evaluated.


SKOPT VS BOTORCH: FUNDAMENTAL DIFFERENCES
------------------------------------------

Both use the same mathematical model (exact GP + EI) but differ in how
they solve it.

1. How the GP linear system is solved (the main scalability difference)
   skopt: exact Cholesky decomposition — computes the full n×n triangular
     factor, O(n^3) time and O(n^2) memory. Non-negotiable cost per fit.
   BoTorch/GPyTorch: conjugate gradient (CG) iterative solver — never
     forms the full Cholesky, solves K·v=b iteratively. Convergence is
     ~O(n) CG steps × O(n^2) per step = O(n^3) asymptotically but with
     much better constants (early stopping, preconditioning). At n=800
     this is ~3–5× faster; the gap widens with n.
   BoTorch can additionally switch to sparse/approximate GPs (SVGP with
     m inducing points) which changes complexity to O(m^2 * n), breaking
     the O(n^3) wall entirely.

2. How the acquisition function is optimised
   skopt: random sampling + simple local search in the original parameter
     space. In 18D this is unreliable — the EI landscape has many local
     optima and random sampling misses most of them.
   BoTorch: gradient-based optimisation via autograd through the GP
     predictive distribution, with num_restarts random starting points.
     More restarts = better chance of finding the global EI maximum.
     This is a meaningful quality difference at 18D — BoTorch's suggested
     next point is genuinely better, not just computed faster.

3. GP hyperparameter fitting
   skopt: scipy L-BFGS-B on the marginal log-likelihood — adequate but
     sometimes gets stuck in local optima.
   BoTorch: PyTorch L-BFGS-B through the same MLL, better numerical
     conditioning via GPyTorch's structured kernel approximations.

4. Normalisation
   skopt: transparent, handled internally.
   BoTorch: manual (written explicitly), which is more code but gives
     full control — useful for debugging unexpected GP behaviour.


WALL-TIME SCALING: SKOPT VS BOTORCH AT HIGH EVAL COUNTS
---------------------------------------------------------
Observed from actual runs (18D, NeuralFoil large model):
  Average ~60s/iter at n~400 (midpoint of 800-iter runs).
  Breakdown: ~10–15s NeuralFoil evaluation (constant) + ~45s GP fitting
  (grows as O(n^3)).

Projected skopt cost per iteration:
  n=400  (observed midpoint): ~60s
  n=800:                      ~375s  (~6 min/iter)
  n=1000:                     ~715s  (~12 min/iter)
  n=2000:                     ~1.5 hrs/iter
  n=5000:                     ~24 hrs/iter  — completely infeasible

Skopt becomes unusable past ~1200–1500 evals for this problem.

BoTorch with approximate GP (inducing points, m~300), O(m^2 * n):
  GP fit stays ~5–10s/iter regardless of n.
  Total per iter dominated by NeuralFoil: ~20–25s.
  5000-eval run: ~28–35 hours wall time — practical on a cluster.

Trigger for switching: skopt is fine up to ~1000 evals. Use BoTorch if
a 3000–6000 eval run is needed for an eval-equivalent comparison with v3.


BOTORCH SETUP (IF NEEDED AFTER SKOPT)
--------------------------------------
BoTorch is the upgrade path if skopt becomes too slow at high eval counts
(typically past ~1,000–2,000 points where exact GP training bogs down).

Installation:
  pip install botorch gpytorch torch

Implementation changes vs skopt (~80–100 lines vs ~30):
  - Replace skopt.Optimizer with botorch.models.SingleTaskGP
  - Fit GP with botorch.fit.fit_gpytorch_mll
  - Optimise acquisition with botorch.acquisition.ExpectedImprovement
    + botorch.optim.optimize_acqf
  - Manually normalise inputs to [0,1] and standardise outputs (skopt
    does this internally; BoTorch requires explicit handling)

Why BoTorch scales better:
  BoTorch uses GPyTorch's approximate GPs (inducing points, CG solvers)
  which scale to tens of thousands of points. skopt uses exact GP
  training which is O(n^3) and degrades past ~1,000-2,000 evals.

Time to implement: ~2-3 hours (mostly normalisation and acq. boilerplate).
Time to run at 6,000 evals: seconds per iteration vs potentially hours for skopt.

Trigger for switching: if skopt runs take too long past ~1,000 evals, or
if a 6,000-eval BO run is needed for an eval-equivalent comparison with v3.
The skopt scaling slowdown is itself a reportable result for the paper.

Reference: Meta's official BoTorch tutorial — SingleTaskGP + ExpectedImprovement.
Ax platform (built on BoTorch) is also widely used in industry.


HOW --gradient_infeasible WORKS (MECHANISM AND DESIGN)
-------------------------------------------------------
The flag is declared in each BO framework's add_args() and piped into the
reward constructor automatically — the framework run() function never reads
it directly.

Mechanism (run_benchmark.py):
  All parsed CLI args are collected into shared_kwargs and passed as **kwargs
  to both reward_class(**shared_kwargs) and env_class(..., **shared_kwargs).
  So --gradient_infeasible, registered by the BO framework, ends up as a
  constructor argument to RewardExactNotebook, which accepts it explicitly.
  The framework run() itself never touches the flag.

Why it is needed for BO but not GA/PSO/v3:
  When NeuralFoil fails to converge on a sample (e.g. degenerate geometry
  that cannot solve all CL targets), the reward is either:

    gradient_infeasible=False (default):
      reward = FAIL_REWARD = -10.0  — a hard constant clip.
      Every infeasible sample looks identical to the GP: flat plateau at -10.
      The GP surrogate has zero gradient signal toward the feasible region.
      BO gets stuck — EI cannot distinguish good-infeasible from bad-infeasible.

    gradient_infeasible=True (BO default):
      reward = fitness_total  — the actual penalty score, which varies with
      how infeasible the design is (e.g. how badly it violates thickness
      constraints). The GP learns that some infeasible regions are "less bad"
      and EI steers toward the feasible boundary rather than wandering.

  GA/PSO/v3 don't build a surrogate model, so a flat landscape in infeasible
  space is harmless — they just treat it as "this design failed" and move on.
  Only BO frameworks need the gradient signal, hence the per-framework default.

Generalisability note:
  For a new environment where all points in parameter space are feasible,
  this flag is irrelevant. It only matters when there is a non-trivial
  infeasible region and the reward hard-clips infeasible samples to a constant.

SCIENTIFIC FRAMING NOTE
-----------------------
GP-BO is the most direct analog to the LLM-guided approach: both are
surrogate-assisted and sample-efficient methods. If the LLM method
outperforms BO, that is a meaningful result for the paper.

Key tension: 18 dimensions is right at the upper edge where GP starts to
struggle. With NeuralFoil being cheap, enough samples can be run to expose
this scaling weakness, which strengthens the argument for LLM-guided methods.


RUNNING HIGH EVAL-COUNT CASES: STRATEGY (March 2026)
-----------------------------------------------------
Recommendation: run n=5000 with the existing exact GP (BO_torch /
SingleTaskGP) first, before implementing approximate GP.

Rationale:
  - BO_torch uses CG solvers (not Cholesky), so it scales as roughly n^1.35
    rather than n^3. Measured on a 128-CPU exclusive node, projected wall
    time for n=5000 is ~35-45 hours — practical on the cluster.
  - Running exact GP to n=5000 first gives a clean, defensible baseline for
    the paper. If approximate GP is introduced later, the two can be directly
    compared on reward quality.
  - The exact GP result also tells you whether n=5000 is actually buying
    meaningful reward improvement over n=1000, before investing further
    engineering effort in approximate GP.

Trigger for approximate GP: if n=5000 exact GP shows continued improvement
and you want to push to n=10000+, or if per-iter wall time degrades past
~60-90s at high n. At n=10000, exact GP is projected at ~180s/iter (~200+
hours total), which is not practical.

Per-iteration wall time projections (exclusive node, 128 CPUs):
  n=1000  end:  ~5-8 s/iter    total: ~1.5-2 hrs
  n=5000  end:  ~70 s/iter     total: ~35-45 hrs
  n=10000 end:  ~180 s/iter    total: ~200-250 hrs  (needs approx GP)

Note: all three n=1000 seeds ran on the same node (no --exclusive), which
caused CPU contention and slower per-iter times. Future runs use --exclusive
so each job gets a full 128-CPU node with no contention.


APPROXIMATE GP IMPLEMENTATION (SVGP WITH INDUCING POINTS)
----------------------------------------------------------
If exact GP becomes too slow past n=5000, the upgrade path is to replace
SingleTaskGP with SingleTaskVariationalGP (BoTorch's SVGP wrapper). This
changes complexity from O(n^3) to O(m^2 * n) where m is the number of
inducing points (m << n), breaking the scaling wall.

Expected per-iter time with approx GP (m~300-500): ~5-10s regardless of n,
making n=10000+ runs practical (~15-30 hours).

Performance impact on reward:
  - In theory: slightly worse, since the surrogate is an approximation.
  - In practice: usually negligible for optimisation (as opposed to
    prediction). The GP only needs to be good enough to guide search toward
    better regions — it doesn't need to be a perfect model.
  - With m=300-500 inducing points for 18D: very close to exact GP quality.
  - At n=5000+, exact GP can develop numerical conditioning issues with
    large covariance matrices; approx GP can actually be more robust.
  - Verdict: do not expect meaningfully worse final reward with well-chosen m.

Code changes vs frameworks/BO_torch/run.py (changes are localised):

  1. Different imports and model class:
       # replace:
       from botorch.models import SingleTaskGP
       from gpytorch.mlls import ExactMarginalLogLikelihood
       model = SingleTaskGP(train_X, train_Y_std)
       mll = ExactMarginalLogLikelihood(model.likelihood, model)

       # with:
       from botorch.models import SingleTaskVariationalGP
       from gpytorch.mlls import VariationalELBO
       model = SingleTaskVariationalGP(train_X, num_inducing=args.num_inducing)
       mll = VariationalELBO(model.likelihood, model.model, num_data=len(train_X))

  2. Different training loop (Adam over epochs, not L-BFGS-B on MLL):
       optimizer = torch.optim.Adam(model.parameters(), lr=0.01)
       for _ in range(100):   # epochs; tune as needed
           optimizer.zero_grad()
           output = model(train_X)
           loss = -mll(output, train_Y_std.squeeze())
           loss.backward()
           optimizer.step()

  3. New argument in add_args():
       parser.add_argument('--num_inducing', type=int, default=300,
                           help='Inducing points for approximate GP (default: 300)')

  4. Inducing point initialisation: SingleTaskVariationalGP picks a random
     subset of train_X by default. K-means initialisation is better but the
     default is usually sufficient.

Implementation approach: create frameworks/BO_torch_approx/ as a separate
framework (do not modify BO_torch). This keeps the exact GP baseline intact
for comparison, and both frameworks produce identical CSV formats so results
are directly comparable in animate_neuralfoil_comparison_eval_axis.py.

Key tuning parameter: num_inducing. Start with 300. If reward quality is
noticeably worse than exact GP on the same budget, increase toward 500.
Diminishing returns past ~500 for an 18D problem.

HOW TO PICK m=300 FOR AN 18D PROBLEM
--------------------------------------
There is no exact formula — it is a rule of thumb built from several
considerations.

1. Coverage of input space
   The inducing points need to be dense enough that no region the GP cares
   about is unseen by any inducing point. In practice BO functions are often
   low effective rank even in high nominal dimension, so m does not need to
   scale with 2^18.

2. Standard heuristic: m ~ 10-20x d
   For d=18: 10x18=180 to 20x18=360. m=300 sits in the middle.
   This range comes from empirical SVGP studies — below ~10d the ELBO
   diverges meaningfully from the exact MLL; above ~20d marginal gain per
   extra inducing point diminishes rapidly.

3. Empirical check: compare against exact GP at the same n
   Run exact and approximate GP on the same ~300-point dataset and check:
     - Do they suggest a similar next point (similar EI landscape)?
     - Is the approximate GP's predicted mean close to the exact GP's at
       held-out points?
   If yes, m is sufficient. If uncertainty estimates are badly off, increase m.

4. Diminishing returns past ~500 for 18D
   Each inducing point adds a row/column to the variational covariance
   S_u in R^(mxm), so cost grows as m^2. Going from m=300 to m=500 roughly
   doubles fit cost but typically buys negligible reward improvement.
   Dropping below m=100 for 18D would likely hurt quality noticeably.

5. Practical wall-time check
   If per-iter GP fit time is already dominated by the NeuralFoil evaluation
   (~10-15s), there is no reason to reduce m. Only reduce m if GP fit time
   starts approaching evaluation time — unlikely at m=300 for 18D.


SVGP PERFORMANCE DIAGNOSIS (March 29, 2026)
--------------------------------------------
After ~4000 evaluations across seeds 0, 1, 3, BO_torch_approx has found
zero feasible solutions. BO_torch (exact GP) achieves 32–61% feasibility
over 2000–2500 evals with best rewards around -0.10 to -0.12. This is a
serious divergence that points to surrogate quality failure, not just
inducing point count.

Root cause: cold-start every iteration
  run.py line 100 rebuilds SingleTaskVariationalGP from scratch on every BO
  iteration — randomly initialised kernel hyperparameters, randomly
  initialised variational parameters, trained for exactly 100 Adam steps at
  lr=0.01. At n=4000 this is severely under-trained. The ELBO landscape for
  SVGP is harder to optimise than exact MLL; 100 steps is unlikely to
  converge even at n=300, let alone n=4000.

  The exact GP (BO_torch) uses L-BFGS-B on the exact MLL, which converges
  reliably to a good hyperparameter estimate each iteration. The approximate
  GP has no such guarantee with fixed-epoch Adam.

  A poorly-converged surrogate gives a noisy EI surface. optimize_acqf then
  finds a spurious EI maximum, effectively exploring near-randomly. This
  explains why the approx runs show barely better than random behaviour
  despite gradient_infeasible=True — the flag only helps if the surrogate
  is actually learning the reward landscape.

Secondary cause: inducing points concentrated in the infeasible region
  When inducing_points=300 is passed as an integer, SingleTaskVariationalGP
  selects a random subset of train_X. Since 0% of evaluations are feasible,
  all 300 inducing points live in the infeasible region. The SVGP has no
  coverage of the feasible region and cannot extrapolate toward it reliably.
  This compounds the cold-start problem.

Key lesson: increasing m does not fix this
  Going from m=300 to m=500 adds more parameters to fit in the same 100
  epochs. It would make convergence worse, not better. The bottleneck is
  training convergence, not surrogate expressiveness.

Recommended fixes in order of impact:

  1. Warm-start the model across iterations
     Carry over kernel hyperparameters and variational parameters from the
     previous iteration's fitted model rather than reinitialising from
     scratch. Each iteration then only needs to update incrementally (new
     observations since last fit). This is the standard trick for
     online/streaming SVGP and can reduce required epochs per iteration
     from ~500 down to ~50 once warm.

  2. Scale n_epochs with n
     100 epochs is insufficient as n grows. A reasonable schedule:
       n_epochs = min(500, 50 + n // 20)
     This keeps early iterations fast and gives later iterations enough
     steps to converge. Combined with warm-starting, even fewer are needed.

  3. K-means inducing point initialisation
     Replace the random subset with k-means cluster centres over train_X.
     This spreads inducing points more evenly across the observed
     distribution. Not sufficient alone (still all infeasible if train_X
     is all infeasible), but better than a random subset.

  4. Learnable inducing point locations
     Pass a coordinate tensor (not an integer) to SingleTaskVariationalGP
     so that inducing point positions are free parameters optimised by the
     ELBO. They can then migrate toward high-information or high-reward
     regions, including the feasible boundary. This is the theoretically
     correct full-SVGP setup.

  5. m=300 is probably fine — fix convergence first
     Do not increase m until warm-starting and epoch scaling are in place.
     Expressiveness headroom is not the current bottleneck.


FIX 1 IMPLEMENTED: WARM-STARTING (March 29, 2026)
---------------------------------------------------
Fix 1 was implemented in frameworks/BO_torch_approx/run.py. Three changes:

  1. Before the loop, initialise prev_state = None.

  2. After building the model each iteration, attempt to load the previous
     state before training:

       if prev_state is not None:
           try:
               model.load_state_dict(prev_state)
           except RuntimeError:
               pass  # shape mismatch while n_inducing is still growing

  3. After model.eval(), save named parameters for the next iteration:

       prev_state = {name: param.detach().clone()
                     for name, param in model.named_parameters()}

How it works:
  named_parameters() snapshots every learnable parameter — kernel
  lengthscales, outputscale, noise, and all variational parameters (means
  and covariances of the variational distribution over inducing outputs).
  On the next iteration, a new model is instantiated (same architecture),
  then each parameter is directly overwritten with its previous value via
  param.data.copy_(). The Adam steps then only need to adjust incrementally
  for the one new observation, rather than converge from random.

  Note: state_dict/load_state_dict was tried first but raised AttributeError
  because SingleTaskVariationalGP's state dict includes training data buffers
  (train_inputs, train_targets) that grow by one entry each iteration.
  Loading them onto a newly constructed model with a different-sized dataset
  caused a shape mismatch inside BoTorch's model internals. Using
  named_parameters() avoids buffers entirely — training data is always set
  correctly by the model constructor and does not need to be warm-started.

Shape guard during inducing point growth phase:
  During the first 300 iterations, n_inducing = len(train_X) (grows from
  31 to 300). Each size change produces differently shaped inducing point
  parameters. The shape check (prev[name].shape == param.shape) silently
  skips mismatched parameters, falling back to cold-start for those. Once
  n >= 300 all shapes are fixed and warm-starting is fully active.

Why BO_torch (exact GP) does not need this:
  BO_torch also cold-starts every iteration (model = SingleTaskGP(...) is
  rebuilt each time), but uses fit_gpytorch_mll which calls L-BFGS-B on the
  exact MLL. L-BFGS-B converges reliably to the MLL optimum in tens of steps
  regardless of initialisation — the exact MLL is a well-conditioned,
  near-convex problem. The ELBO (SVGP objective) is harder and non-convex,
  so Adam from random initialisation cannot converge in 100 steps at large n.

Seeds rerun after fix: 0, 1, 3, 42 (jobs 509–512, submitted March 29 2026).
Jobs crashed immediately with AttributeError on train_targets; state_dict
approach replaced with named_parameters. Resubmitted as jobs 513–516, then
513–516 also crashed for the same reason (cached results from prior run caused
jobs to skip to iter 31 where warm-start was first attempted). Results cleared,

FIX 3 IMPLEMENTED: K-MEANS INDUCING POINT INITIALISATION (March 30, 2026)
---------------------------------------------------------------------------
Observation that motivated Fix 3:
  With Fix 1 (warm-starting) in place, incumbent improved rapidly during
  the random phase and early BO (n < 300), then stalled completely at n=300.
  This is the exact point where n_inducing hits its cap of 300 and the model
  switches from using all training points as inducing points (effectively
  exact GP) to a true approximation with a fixed subset. The random subset
  (default) placed all 300 inducing points in the infeasible region, giving
  no surrogate coverage of the feasibility boundary.

Implementation:
  Replaced the integer inducing_points=n_inducing argument with a k-means
  coordinate tensor. When n > n_inducing, MiniBatchKMeans clusters train_X
  into n_inducing centres; when n <= n_inducing all training points are used
  (k-means would be one cluster per point, so we skip it and use train_X
  directly).

    if n_inducing >= len(train_X):
        inducing_pts = train_X.clone()
    else:
        from sklearn.cluster import MiniBatchKMeans
        km = MiniBatchKMeans(n_clusters=n_inducing, n_init=3,
                             random_state=args.random_state)
        km.fit(train_X.numpy())
        inducing_pts = torch.tensor(km.cluster_centers_, dtype=torch.float64)

    model = SingleTaskVariationalGP(
        train_X, train_Y=train_Y_std, inducing_points=inducing_pts,
    )

  When a coordinate tensor is passed, GPyTorch's VariationalStrategy
  registers inducing_points as nn.Parameter, so Adam optimises their
  locations during training (Fix 4 comes for free).

Result:
  Incumbent improved to -3.4 (seed0) and -5.3 (seed1) by n=300 — better
  than the Fix-1-only runs at the same point. However, both seeds stalled
  completely from n=300 to n=1100+, just as with Fix 1 alone. The stall at
  n=300 is robust across both random-subset and k-means initialisation,
  pointing to epoch count (Fix 2) rather than placement as the bottleneck
  once the model is truly approximate.

Seeds rerun after fix: 0, 1, 3, 42 (jobs 523–526, submitted March 30 2026).


FIX 2 IMPLEMENTED: EPOCH SCALING WITH DATASET SIZE (March 30, 2026)
---------------------------------------------------------------------
Motivation:
  Both Fix 1 and Fix 1+3 showed a sharp stall at exactly n=300 — the point
  where n_inducing hits its cap. Before n=300, n_inducing = n (all points
  as inducing points, effectively exact GP), and 100 epochs is sufficient.
  Once n > 300 and the model is a true SVGP approximation, the ELBO
  landscape grows more complex with each new observation, and 100 epochs
  with warm-starting is no longer enough to converge the variational
  parameters.

Implementation:
  Replaced the fixed args.n_epochs with a dynamic schedule computed each
  iteration. args.n_epochs (default 100) acts as a floor; the schedule
  scales above it as n grows, capped at 500.

    n_epochs = min(500, max(args.n_epochs, len(train_X) // 3))

  Schedule values:
    n=300:   max(100, 100) = 100 epochs  (same as before, at transition point)
    n=400:   max(100, 133) = 133 epochs  (scaling kicks in immediately after)
    n=600:   max(100, 200) = 200 epochs
    n=1000:  max(100, 333) = 333 epochs
    n=1500:  max(100, 500) = 500 epochs  (cap reached)
    n=10000: min(500, ...)  = 500 epochs

  The original formula (50 + n // 20) was found to not kick in until n=1000
  since it only exceeds args.n_epochs=100 when n > 1000. The stall occurs at
  n=300, so the scaling needed to activate immediately after that point.
  n // 3 achieves this — it equals 100 at n=300 and grows from there.

  The floor at args.n_epochs means the default behaviour is unchanged at
  small n, and the user can still raise the minimum via --n_epochs if needed.
  The cap at 500 bounds wall-time cost; with warm-starting, 500 epochs for
  incremental updates is acceptable even at large n.

Result (fixes 1+2+3 combined, jobs 561–564 then 566–569, March 30 2026):
  Seeds 0, 1, 3 all stalled at n=300 or shortly after. Seed0 completely
  flat at -3.356 from n=301 to n=700+. Seed1 made one small jump then
  stalled. Seed3 improved slowly to -5.1 but plateaued. Zero feasible
  solutions across all seeds even with all three fixes active.
  Conclusion: the stall is not primarily a coverage or convergence issue —
  it is a fundamental SVGP approximation quality problem on this specific
  problem type. Fixes were stopped at this point.


REMAINING SVGP FIXES NOT IMPLEMENTED (decision: treat as negative result)
--------------------------------------------------------------------------
Three further fixes were identified but not implemented after fixes 1-3
failed to break the n=300 stall. Recorded here for completeness.

Fix 4 — Reward clipping
  Infeasible rewards range from -30 to -100. After standardisation, these
  extreme values dominate and the surrogate spends capacity modelling deep
  infeasibility rather than the region near the feasibility boundary.
  Clipping to a floor (e.g. -15) before standardisation would compress the
  infeasible landscape and give the surrogate more resolution near where it
  matters. One-line change: train_Y_clipped = train_Y.clamp(min=-15.0).

Fix 5 — Natural gradient descent (NGD)
  Adam treats all parameters equally, but variational parameters have a
  different geometric structure — NGD accounts for this via the Fisher
  information metric and is the theoretically correct optimiser for
  NaturalVariationalDistribution. In practice NGD often converges in 10-20
  steps where Adam needs hundreds. Implementation requires replacing
  SingleTaskVariationalGP with a custom _NaturalSVGP class (GPyTorchModel +
  ApproximateGP multiple inheritance), using NaturalVariationalDistribution,
  and two separate optimisers: NGD(model.variational_parameters(), lr=0.1)
  and Adam(model.hyperparameters(), lr=0.003).

Fix 6 — Lower Adam learning rate
  lr=0.01 may overshoot with warm-starting since the model is already near
  convergence each iteration. Dropping to lr=0.003 for incremental updates
  is a common SVGP tuning recommendation.

Why these were not pursued:
  After fixes 1-3, seed0 was flat for 400+ iterations despite warm-starting,
  k-means coverage, and scaled epochs. The remaining fixes are increasingly
  speculative and each requires another debugging cycle (the NGD rewrite in
  particular introduces new failure modes). The exact GP at n=3000-5000 is
  already producing strong, defensible results (-0.096 best reward, 33-65%
  feasibility). The SVGP failure is itself a reportable negative result.


SVGP: WHEN IT WOULD WORK BETTER
---------------------------------
The NeuralFoil problem is close to a worst case for SVGP-BO. For reference,
the conditions under which SVGP-BO tends to work well:

1. Smooth, continuous reward landscape
   This problem has a sharp feasibility boundary — designs jump from deeply
   infeasible (reward -10 to -100) to feasible (reward -0.1 to -0.3) with
   no gradient signal in between (before gradient_infeasible is applied).
   SVGP works much better when the reward varies smoothly everywhere and
   there is no discontinuous boundary to locate.

2. Large feasible region
   The feasible region here is a small island in 18D space requiring 7
   simultaneous constraints to be satisfied. Random search finds ~0% feasible
   designs, so the SVGP trains entirely on infeasible data for hundreds of
   iterations. SVGP works better when random search finds 20-50% feasible
   designs immediately — the surrogate has a mixed training set from the
   start and can model the landscape on both sides of the boundary.

3. Lower effective dimensionality (d < 10)
   m=300 inducing points covers 18D space poorly — the ratio m/d=17 is at
   the lower end of the recommended 10-20x heuristic, and coverage degrades
   exponentially with dimension. The same SVGP on a 5-8D problem would have
   far better relative coverage and would almost certainly outperform the
   exact GP at high n.

4. Large n genuinely needed (n >> 5000)
   SVGP earns its keep when you need n=10000+ and the exact GP is the
   computational bottleneck. If the exact GP at n=3000-5000 already
   saturates reward improvement — as appears to be the case here — there
   is no motivation for SVGP in the first place.

5. High feasibility from random search
   Corollary to point 2: problems where the constraint landscape is weak
   or absent (unconstrained optimisation, or soft constraints only) are
   ideal. SVGP does not need to locate a feasibility boundary at all.
