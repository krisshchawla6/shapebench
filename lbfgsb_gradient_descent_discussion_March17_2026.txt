Gradient-Based Optimization in ShapeEvolve: L-BFGS-B Formulation
=================================================================
Date: March 17, 2026


CONTEXT
-------
ShapeEvolve benchmarks LLM-guided evolutionary design against classical
optimizers. The existing classical baseline is PSO (frameworks/GA/). This
document describes how to add an adjoint/gradient-based baseline using
L-BFGS-B via scipy.optimize.minimize, mirroring the PSO structure exactly.


PROBLEM FORMULATION
-------------------
maximize  f(x)  =  reward(x)      [e.g. CL/CD at alpha=5 deg for ld_ratio]
subject to  lb <= x <= ub         [18 Kulfan CST parameter bounds]

Design vector x (18 parameters):
  upper_weights[0:8]      upper-surface CST coefficients   bounds (-0.30,  0.60)
  lower_weights[8:16]     lower-surface CST coefficients   bounds (-0.30,  0.30)
  leading_edge_weight[16] leading-edge modification        bounds (-0.50,  0.50)
  TE_thickness[17]        trailing-edge thickness          bounds ( 0.000, 0.010)

Baseline geometry: approx. NACA 4412.


WHY L-BFGS-B OVER PLAIN GRADIENT DESCENT
-----------------------------------------
The implementation described in the initial design discussion used plain
gradient ascent with a fixed learning rate. This is non-standard for three
reasons:

1. No momentum / adaptive step size.
   Standard practice uses Adam (adaptive per-parameter lr + momentum) or a
   quasi-Newton method. A fixed lr either overshoots or creeps depending on
   local curvature.

2. No line search.
   Without a line search, convergence depends entirely on lr being well-tuned.
   L-BFGS-B performs an internal Wolfe-condition line search every iteration,
   making it robust without hyperparameter tuning.

3. No Hessian approximation.
   L-BFGS-B accumulates a low-rank approximation of the inverse Hessian from
   gradient history. This gives Newton-like convergence rates (superlinear near
   the optimum) without the O(n^2) cost of computing the full Hessian.

Gradient computation: finite differences (3-point, jac='3-point' in scipy).
Cost per iteration: 2*18 = 36 simulator calls for the gradient + line search
calls. For NeuralFoil (in-process NN inference) this is negligible. For heavier
solvers (FEniCS, CFD) a true adjoint would be needed.

True adjoint vs finite differences:
  - Finite differences: O(n) solver calls per gradient, straightforward to
    implement, works with any black-box simulator.
  - True adjoint: O(1) solver calls per gradient (cost independent of n),
    requires differentiating through the solver (e.g. CasADi in AeroSandbox,
    or AD through a PyTorch/JAX surrogate). NeuralFoil/AeroSandbox supports
    CasADi autodiff internally, so true adjoints are possible by calling the
    solver directly rather than through write_design/simulate.

scipy.optimize.minimize with method='L-BFGS-B' handles bounds, line search,
and the quasi-Newton approximation out of the box. No custom optimizer loop
is needed.


WHY L-BFGS-B OVER PURE FINITE DIFFERENCES OR A TRUE ADJOINT
-------------------------------------------------------------
These are three separate concerns that are often conflated:
  - How you COMPUTE the gradient       (finite differences vs adjoint)
  - How you USE the gradient           (plain gradient descent vs L-BFGS-B)
  - How you CONSTRAIN the search       (projection vs bound-aware method)

L-BFGS-B is a choice about (2) and (3). It is orthogonal to (1): you can
feed L-BFGS-B a gradient computed by either finite differences or a true
adjoint. The comparison below addresses each pairing.

L-BFGS-B + finite differences vs plain gradient descent + finite differences:
  Pure finite-difference gradient descent gives you the gradient direction but
  throws away curvature information. Every step costs the same 2n simulator
  calls regardless of how close you are to the optimum, and you must manually
  tune a learning rate that works across the whole trajectory. L-BFGS-B uses
  the same finite-difference gradient calls but additionally builds a running
  approximation of the inverse Hessian from the history of (x, grad) pairs.
  This means:
    - Steps are scaled by local curvature, so flat directions take larger steps
      and steep directions take smaller ones automatically.
    - The internal Wolfe line search selects the step length each iteration,
      eliminating the learning rate hyperparameter entirely.
    - Convergence is superlinear near the optimum (quasi-Newton) vs linear for
      plain gradient descent. In practice this means far fewer iterations to
      reach the same objective value.
  The extra cost per iteration is negligible (storing ~10 vector pairs for the
  Hessian approximation). There is almost no reason to use plain gradient
  descent over L-BFGS-B when the simulator is cheap and scipy is available.

L-BFGS-B + finite differences vs L-BFGS-B + true adjoint:
  Switching from finite differences to a true adjoint does not change the
  optimizer at all -- L-BFGS-B receives a gradient vector either way. The
  difference is purely in gradient cost:
    - Finite differences: 2n simulator calls per gradient (36 for 18 params).
      Black-box: works with any simulator without modification.
    - True adjoint: ~1-2 solver passes per gradient regardless of n. Requires
      either differentiating through the solver (CasADi, JAX, PyTorch autograd)
      or deriving and implementing the adjoint equations by hand.
  For NeuralFoil specifically, the simulator is fast enough that 36 calls per
  gradient is not a bottleneck, so finite differences is the pragmatic choice.
  The adjoint becomes worthwhile when:
    (a) n is large (hundreds of shape parameters or more),
    (b) the simulator is expensive (FEniCS, full CFD), or
    (c) you need gradients to machine precision rather than O(eps) accuracy.
  AeroSandbox's CasADi backend would allow a true adjoint for NeuralFoil
  without rewriting the physics, but it would require bypassing the
  write_design/simulate interface and calling the solver directly.

Summary table:

  Method                         | Gradient cost | Tuning required | Convergence
  -------------------------------|---------------|-----------------|------------
  GD + finite diff               | O(n) calls    | learning rate   | linear
  L-BFGS-B + finite diff         | O(n) calls    | none            | superlinear
  L-BFGS-B + true adjoint        | O(1) calls    | none            | superlinear
  L-BFGS-B + true adjoint (CFD)  | O(1) calls    | none            | superlinear

  Recommended for NeuralFoil: L-BFGS-B + finite differences.
  Recommended for FEniCS/CFD:  L-BFGS-B + true adjoint.

CODE: frameworks/adjoint/gradient_descent/gradient_descent.py
--------------------------------------------------------------
"""L-BFGS-B optimizer wrapper -- thin shim around scipy.optimize.minimize."""

import numpy as np
from scipy.optimize import minimize


class LBFGSOptimizer:
    """Gradient ascent via L-BFGS-B (minimizes -reward).

    Parameters
    ----------
    x0 : np.ndarray   Initial design vector.
    lb, ub : np.ndarray   Per-parameter bounds.
    maxiter : int     Maximum L-BFGS-B iterations.
    eps : float       Finite-difference step for gradient estimation.
    """

    def __init__(self, x0, lb, ub, maxiter=200, eps=1e-4):
        self.x0      = x0.copy()
        self.bounds  = list(zip(lb.tolist(), ub.tolist()))
        self.maxiter = maxiter
        self.eps     = eps

    def run(self, reward_fn, callback=None):
        """Run the optimizer.

        Parameters
        ----------
        reward_fn : callable  x -> float  (higher is better)
        callback  : callable  invoked after each iteration with the
                              current x and negated-reward value.

        Returns
        -------
        result : scipy.optimize.OptimizeResult
        """
        def objective(x):
            return -reward_fn(x)   # L-BFGS-B minimizes

        result = minimize(
            objective,
            self.x0,
            method='L-BFGS-B',
            jac='3-point',         # finite-difference gradient (2n calls)
            bounds=self.bounds,
            options={
                'maxiter': self.maxiter,
                'eps':     self.eps,
                'ftol':    1e-9,
                'gtol':    1e-6,
            },
            callback=callback,
        )
        return result


CODE: frameworks/adjoint/run.py
-------------------------------
"""Adjoint framework entry point -- L-BFGS-B via scipy."""

import os
import csv
import numpy as np

from frameworks.core.database import update_database, empty_database
from frameworks.adjoint.gradient_descent.gradient_descent import LBFGSOptimizer


def add_args(parser):
    parser.add_argument('--maxiter', type=int, default=200,
                        help='Max L-BFGS-B iterations (default: 200)')
    parser.add_argument('--eps', type=float, default=1e-4,
                        help='Finite-difference step size (default: 1e-4)')
    parser.add_argument('--n_restarts', type=int, default=1,
                        help='Number of random restarts (default: 1)')


def run(environment, args, output_dir):
    os.makedirs(output_dir, exist_ok=True)
    lb, ub = environment.get_param_bounds()

    csv_path  = os.path.join(output_dir, 'results.csv')
    base_cols = ['iteration', 'restart', 'reward', 'best_reward']
    env_cols  = environment.get_results_csv_columns()
    with open(csv_path, 'w', newline='') as f:
        csv.writer(f).writerow(base_cols + env_cols)

    database      = empty_database()
    global_best   = -np.inf
    global_best_x = None
    call_counter  = [0]   # mutable for closure

    for restart in range(args.n_restarts):
        x0 = lb + np.random.rand(len(lb)) * (ub - lb)
        opt = LBFGSOptimizer(x0, lb, ub, maxiter=args.maxiter, eps=args.eps)

        def reward_fn(x):
            t           = call_counter[0]
            case_dir    = os.path.join(output_dir, f'call_{t:05d}_r{restart}')
            design_path = environment.write_design(x, case_dir, 'design')
            reward, results = environment.simulate(design_path, case_dir)
            database_ref = update_database(database, design_path, reward, results)

            nonlocal global_best, global_best_x
            if reward > global_best:
                global_best   = reward
                global_best_x = x.copy()

            metrics = results.get('metrics', {}) if results else {}
            with open(csv_path, 'a', newline='') as f:
                csv.writer(f).writerow(
                    [t, restart, f'{reward:.6f}', f'{global_best:.6f}']
                    + environment.get_results_csv_row(metrics)
                )

            call_counter[0] += 1
            return reward

        result = opt.run(reward_fn)
        print(f'[L-BFGS-B] restart {restart+1}/{args.n_restarts}  '
              f'best={global_best:.4f}  '
              f'converged={result.success}  msg={result.message}')

    print(f'[L-BFGS-B] Done. Global best reward: {global_best:.6f}')

DESIGN NOTES
------------
- gradient_descent.py is a pure math object: knows nothing about airfoils,
  only operates on arrays. Could optimize any bounded continuous problem.

- run.py is the adapter: maps the environment's file-based interface
  (write_design / simulate) onto the optimizer's reward_fn callable.

- call_counter is a list rather than an int so it is mutable inside the
  reward_fn closure (Python closures cannot rebind nonlocal ints without
  the nonlocal keyword; using a list avoids that for the counter).

- n_restarts > 1 is recommended because L-BFGS-B is a local optimizer and
  will converge to different local optima from different starting points.
  The global best across all restarts is tracked in global_best.

  Why this matters -- multimodal reward landscapes:
  L-BFGS-B follows the gradient uphill from its starting point and stops at
  the nearest local optimum. It has no mechanism to escape once converged.
  The Kulfan CST parameter space is likely multimodal: a thin low-camber
  shape and a higher-camber shape can both be locally optimal for L/D at a
  given alpha, and L-BFGS-B started near one will never find the other.

  What restarts do:
  Each restart draws a new x0 uniformly from the bounds:
      x0 = lb + np.random.rand(len(lb)) * (ub - lb)
  L-BFGS-B then converges to whichever local optimum is in that starting
  point's basin of attraction. With enough restarts you sample different
  basins and take the global best across all runs -- a crude but effective
  approximation of global search.

  Comparison to true global optimizers:
    Method                    Global search mechanism        Cost
    ------------------------  -----------------------------  -------------------------
    L-BFGS-B x1              None                           Fast, likely suboptimal
    L-BFGS-B x k restarts    Random starting points         k* cost, better coverage
    CMA-ES                    Population + cov. adaptation   Built-in, more principled
    DE                        Population + crossover/mut.    Built-in, strong global

  CMA-ES and DE have global exploration built into their algorithms. L-BFGS-B
  with restarts is a poor man's version: independent local runs hoping to
  cover the space by luck.

  Fair comparison with LLM frameworks:
  The iteration counter in results.csv tracks total simulator calls across
  all restarts, not per-restart iterations. L-BFGS-B (3 restarts x 200
  iterations) and an LLM islands run with the same call budget are therefore
  directly comparable -- both used the same number of NeuralFoil evaluations.
  The question is whether L-BFGS-B's gradient efficiency within each restart
  beats the LLM's structured exploration across the same total budget.

  Practical guidance: run enough restarts that global_best stabilizes across
  different random seeds. If it keeps improving with each additional restart,
  you have not run enough.

- The 'iteration' column in results.csv counts total simulator calls across
  all restarts, not L-BFGS-B iterations, so it is directly comparable to
  the LLM call count used by the islands/v2 frameworks.


TRUE ADJOINT WITH NEURALFOIL: IS IT POSSIBLE?
----------------------------------------------
Yes -- and NeuralFoil is one of the cleanest cases where a true adjoint is
achievable without modifying the physics code at all.

NeuralFoil uses `aerosandbox.numpy` throughout its forward pass rather than
standard numpy. AeroSandbox's numpy is a drop-in replacement that dispatches
to CasADi symbolic operations when the inputs are CasADi variables, and to
regular numpy otherwise. This means the entire neural network forward pass --
matrix multiplications, activations, everything -- is automatically
differentiable via CasADi's symbolic AD engine with no changes to NeuralFoil.

To get a true adjoint gradient you bypass write_design/simulate entirely and
call the solver directly with CasADi symbolic variables:

    import casadi as ca
    from neuralfoil import get_aero_from_kulfan_parameters

    # Design vector as CasADi symbolics
    x = ca.MX.sym('x', 18)

    kulfan = {
        "upper_weights":       x[:8],
        "lower_weights":       x[8:16],
        "leading_edge_weight": x[16],
        "TE_thickness":        x[17],
    }

    aero   = get_aero_from_kulfan_parameters(kulfan, alpha=5.0, Re=1e6)
    reward = aero["CL"] / aero["CD"]

    # CasADi builds the gradient symbolically -- one pass, exact
    grad_fn = ca.Function('grad', [x], [ca.gradient(reward, x)])

grad_fn(x0) then returns the exact gradient in a single forward+backward
pass, independent of n=18. Compare this to finite differences which requires
2*18 = 36 forward passes.

This gradient can then be passed directly to L-BFGS-B by replacing
jac='3-point' with a callable:

    def adjoint_grad(x):
        return np.array(grad_fn(x)).flatten()

    result = minimize(objective, x0, method='L-BFGS-B',
                      jac=adjoint_grad, bounds=bounds)

Practical caveats:
  - The clip calls in NeuralFoil's _sigmoid use aerosandbox.numpy's np.clip,
    which is CasADi-compatible, so the network itself traces cleanly.
  - The reward functions (e.g. constrained_ld, multipoint_hpa) also need to
    be CasADi-safe. Simple arithmetic (CL/CD, penalties) traces fine. Any
    Python control flow that branches on the value of a CasADi variable
    (e.g. if reward > threshold) will not trace and would need replacing with
    ca.if_else().
  - The multipoint_hpa reward uses Brent's method bisection to solve for
    alpha at each CL target, which is iterative and not directly
    differentiable. A true adjoint for that reward would require either
    implicit differentiation through the root-finding step or replacing
    Brent's method with a CasADi-compatible Newton solve.
  - For ld_ratio and constrained_ld the entire pipeline from x to reward
    is smooth and CasADi-traceable with no modifications.

Summary:
  - ld_ratio, constrained_ld, max_cl: true adjoint is straightforward.
  - multipoint_hpa: requires additional work to handle the bisection solver.
  - The adjoint approach requires stepping outside the write_design/simulate
    interface, so it would not plug into run.py as-is. It would need its own
    run.py that calls the solver directly rather than through the environment.
