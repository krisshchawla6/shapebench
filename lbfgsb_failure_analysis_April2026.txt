WHY THESE ARE TRUE FAILURES (unlike v3 premature cancellations)
---------------------------------------------------------------
The key distinction from v3 dynamic optimizer failures is that L-BFGS-B is
a gradient-based, deterministic optimizer. This has important consequences:

1. ALL 3 RESTARTS COMPLETED
   Every failed case ran all 3 restarts to convergence (gradient ≈ 0).
   These were not early cancellations — the optimizer ran its full budget
   and converged. Compare to v3 flash2_5 "failures" where attempts 11, 17,
   19 were cancelled before their budget expired.

2. DETERMINISTIC CONVERGENCE TO LOCAL MINIMA
   Each restart follows the gradient from its random starting point and
   converges to a local minimum. Once converged, the optimizer stops —
   it has no mechanism to escape. Running additional iterations from an
   already-converged point would do nothing.

3. NO STOCHASTIC ESCAPE MECHANISM
   In v3, the LLM generates new candidate designs at every iteration,
   creating the possibility of a random jump to a feasible region at any
   time. L-BFGS-B has no equivalent — it is purely gradient-driven. A
   run that converges to an infeasible local minimum cannot recover without
   a new restart from a different starting point.

4. FAILURE IS SEED-DETERMINED
   Given the same seed and same environment, a failed restart will always
   produce the same result. There is no scenario where "running longer"
   would help — the outcome is fixed by the starting point and the local
   landscape.

WHAT THE best_reward VALUES INDICATE
-------------------------------------
Most failed cases show best_reward = -1.000 or similar round negative values
(e.g. -2.000, -3.000). These correspond to the optimizer converging to designs
where a fixed number of CL targets are unreachable — the reward reflects the
penalty structure, not a near-miss on feasibility. There is no gradient signal
pointing toward feasibility from those local minima.

COULD MORE RESTARTS HELP?
--------------------------
Yes — the failed seeds failed because all 3 of their random starting points
landed in infeasible basins of attraction. Running the same seeds with nr > 3
(e.g. nr10) would give each seed more chances to start in a feasible basin.
However, this is a different experiment (more compute budget), not a question
of whether the existing runs were cut short. Within the defined nr3 budget,
these are unambiguous failures.

NR10 RETRY RESULTS (April 2026)
---------------------------------
All 22 nr3-failed seeds were rerun with nr10 (--random_state identical, so
restarts 0-2 are the same starting points as nr3; restarts 3-9 are additional
draws from the same seeded sequence).

Most previously-failed seeds found feasibility with the extra restarts.
A small number of stubborn seeds (8, 14, 30, 31) remained all-infeasible or
near-infeasible even at 10 restarts, suggesting very low per-restart success
probability (p) for those seeds in the 18D landscape.

WOULD NR > 10 YIELD FURTHER BENEFIT?
--------------------------------------
Diminishing returns — depends on p (per-restart probability of landing in a
feasible basin):

  - If p is moderate (30-50%): nr10 already gives 97-99.9% success probability
    (1 - (1-p)^10). Going to nr20 adds little.
  - If p is very low (5-10%): nr10 gives only 40-65% success probability.
    The stubborn seeds (8, 14, 30, 31) likely fall in this category.
    nr20 would give 64-88%, nr40 would give 87-99% — meaningful gains,
    but at high compute cost for just a few seeds.

Practical recommendation: nr10 is sufficient for most seeds. For the handful
that remain stuck at nr10, the marginal value of further retries depends on
whether a complete 40-seed dataset is required. For typical use (best-result
comparison, statistical summary), the nr10 recoveries plus original nr3
successes provide a large enough sample.
