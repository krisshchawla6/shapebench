DrivAer_Star Benchmark Discussion
April 9, 2026
================================================================================

OVERVIEW
--------
Goal: Benchmark four optimization methods (GA/PSO, L-BFGS-B, BO_torch, v3 LLM)
on a 3D aerodynamic shape optimization problem: minimize drag coefficient (Cd) of
a car body (DrivAer geometry) parameterized by 20 FFD (Free-Form Deformation)
parameters. No constraint on lift — objective is Cd only.

Operating conditions: rho=1.25 kg/m^3, U=40 m/s, area_ref=2.37 m^2.
Reward: reward = -Cd  (maximize reward = minimize drag).

SURROGATE: Transolver — a transformer-based neural network surrogate that predicts
drag and lift from the deformed surface mesh. Input: ~50,000 surface cells with
cell centers, areas, and normals (7 features per cell). Output: Cd, Cl.

Unlike NeuralFoil, Transolver has no analysis_confidence metric and the pipeline
involves a non-differentiable VTK mesh geometry step — so no adjoint/IPOPT
post-processing is possible. Raw optimizer results are the final benchmark outputs.
(See drivaer_star_adjoint_feasibility_April_8_2026.txt for detailed analysis.)

PERFORMANCE OPTIMIZATIONS APPLIED (April 8, 2026)
--------------------------------------------------
Two changes made to DrivAer_Star environment before submitting jobs:

  1. Image rendering disabled (render_images=False default). Eliminated the
     dominant ~20-60 s/eval overhead from off-screen PyVista rendering.

  2. Base VTK mesh cached in memory (_BASE_MESH_CACHE dict). pv.read() called
     once per process; subsequent calls use .copy() on the cached mesh.
     Saves ~1-3 s/eval.

Together: per-eval cost from ~23-70 s → ~3-10 s (CPU). Measured on running
L-BFGS-B jobs: 4.94 s/eval (seed19, 13120 evals over ~18h).


WHY L-BFGS-B IS SLOW
---------------------
The bottleneck is 3-point finite-difference gradients in 20 dimensions:
  - Each gradient step: 2 × 20 = 40 extra forward evaluations
  - At ~5 s/eval: ~3.3 min per gradient, ~3.75 min per L-BFGS-B iteration
  - maxiter=200 × 3 restarts → ~37h total per seed (theoretical max)

Compare to NeuralFoil L-BFGS-B: microseconds per eval → full 3-restart run
in minutes. The eval cost, not the algorithm, is the bottleneck here.

CONVERGENCE ANALYSIS:
18/20 seeds fully converged — all completed all 3 restarts and best Cd flatlined
well before the end of the run (many as early as 25% through). Several seeds
(e.g. 11, 12, 13, 17) found their local minimum and never improved again after
the first quarter of evaluations. The Cd variance across seeds (0.066–0.093)
reflects the distribution of local minima in the 20D landscape, not insufficient
iterations — it is purely initialization-dependent.

2 exceptions:
  seed1:  Δ=0.00327 in last 10% — still improving at termination. Started in an
          extremely poor basin (Cd~0.23) and spent the entire run climbing out
          (0.229 → 0.185 → 0.093). Even with more iterations it is unlikely to
          reach the quality of well-initialized seeds.
  seed3:  Δ=0.00011 in last 10% — marginal, effectively converged.

Running L-BFGS-B longer would not help: the 18 converged seeds have genuinely
found their local optima, and seed1's trajectory is a consequence of poor
initialization that restarts could not overcome.

V3 LLM — NaN PARAMETER BUG (discovered April 12, 2026)
--------------------------------------------------------
During analysis of the n10000 runs, design JSON files were found containing
literal `NaN` values for FFD parameters (e.g. `"side_mirrors_z": NaN`).
NaN is not valid per the JSON spec but Python's json module reads it silently,
so the framework accepted and saved these corrupted files.

Scale of the problem (across 20 n10000 runs):
  side_mirrors_z:       672 NaN occurrences  (anomalously high — ~5× others)
  All other 19 params:  ~117–131 each
  Total corrupted evaluations: ~2,540 / ~200,000 total (~1.3% of budget)

Consequences:
  - Most NaN-param designs → reward=NaN, Cd=NaN in results.csv. The surrogate
    evaluated them, produced invalid geometry, and recorded NaN. Those budget
    slots are wasted.
  - A minority were silently evaluated with NaN treated as 0 by the surrogate
    (e.g. iter_260_o3: NaN in side_mirrors_z but Cd=0.223 recorded). Those
    evaluations ran on a different design than intended.
  - The best designs from v3 are all clean (no NaN) — best Cd is unaffected.
    The n10000 best Cd (0.06516 mid-run) is from a clean design.

Root cause: the LLM uses `NaN` as an implicit "no strong opinion / keep default"
signal for parameters it is uncertain about, instead of a concrete float. The
`side_mirrors_z` parameter is particularly susceptible — the model likely has poor
calibration on its narrow ±0.05 m range relative to the ±8° angular parameters.

Fix: v3 framework should validate all design parameters before calling the
environment (clamp to bounds or resample if NaN/Inf). This is a known failure
mode of structured LLM output: when forced to produce JSON with numeric values,
models occasionally emit NaN rather than a plausible number.

Impact on analysis: NaN-param designs are excluded from all analysis scripts
(convergence curve uses best_reward column which is NaN for failed evals and
thus ignored by the running-max logic; PCA/radar scripts explicitly filter them).

AERODYNAMIC CHARACTERISTICS OF THE THREE BODY STYLES (from DrivAerStar paper)
  The three body styles have meaningfully different Cd distributions across the
  sampled design space (Qiu et al., NeurIPS 2024 / arXiv:2510.16857):

  Notchback (N): unimodal distribution, μ ≈ 0.28, narrow IQR.
    The most well-behaved of the three — a single aerodynamic regime dominates,
    and sampled designs cluster tightly around the mean. BO results should be
    consistent and easy to interpret.

  Fastback (F): right-skewed distribution, μ ≈ 0.33.
    Most designs cluster near the mean but a tail of high-drag configurations
    exists. The skew suggests there are some parameter directions that
    disproportionately increase drag. BO should handle this well given its
    acquisition function's ability to exploit the low-drag tail.

  Estateback (E): BIMODAL distribution, density peaks at Cd ≈ 0.31 and ≈ 0.35.
    This is the critical outlier. The two peaks indicate two distinct
    aerodynamic regimes — most likely corresponding to different wake
    attachment states (attached vs. separated flow over the rear deck), which
    is a well-known feature of estate/notchback geometries near the critical
    ramp angle. Designs can switch between regimes with relatively small
    parameter changes.

  Implications for the Estate benchmark results:
    - The bimodal regime structure is consistent with the high variance across
      BO seeds observed in the Estate benchmark (range 0.0651–0.0658 is tight
      within-regime, but the variance ACROSS seeds that end in different basins
      would be wider if n=1000 were insufficient to escape the poor regime).
    - The bounds-convergence finding (car_size=0.800, ramp_angle≈+8°) likely
      corresponds to one specific aerodynamic regime — the optimizer has found
      and locked onto the low-drag basin. Designs starting in the high-drag
      basin (Cd≈0.35) may need more evaluations to cross the regime boundary.
    - For paper presentation: the Estate body is not a "typical" test case —
      its bimodal landscape makes it harder than F or N. Results on E may not
      generalise directly to either body style. Reporting all three separately
      and noting the distributional differences is important.
    - The bimodality also means that surrogate accuracy metrics (MAPE=2.64%)
      averaged over both regimes may mask higher errors near the regime
      boundary, where small geometry changes cause large Cd jumps.

  Note on Cd scale: the dataset distribution values (μ≈0.28–0.33) are higher
  than our surrogate baseline Cd values (E=0.223, F=0.190, N=0.181) because
  the dataset samples many FFD deformations that increase drag, while our base
  mesh is a specific design that happens to lie below the distribution mean.

BASELINE Cd AS REFERENCE — IMPLICATIONS (April 15, 2026)
---------------------------------------------------------
  The zero-param base mesh Cd (E=0.223, F=0.190, N=0.181) is below the
  dataset distribution mean (0.28–0.33 depending on body style). This means
  the original DrivAer geometry is already a better-than-average design within
  the FFD design space — it was engineered to be aerodynamically reasonable,
  so it sits in the lower half of the distribution of randomly sampled variants.

  Implications for the benchmark:

  1. Improvement metric framing.
     Reporting percentage reduction from baseline (e.g. E: 0.223→0.065, ~71%)
     is technically correct but may be read as measuring from a "naive" or
     typical starting point. It is not — the baseline is already below the
     dataset mean. A reader comparing against other benchmarks should be aware
     that the absolute reduction from the dataset mean would be larger (~77%
     from μ≈0.28 for N, ~80% from μ≈0.33 for F) and arguably a more
     representative "improvement over a random design" figure.
     Recommendation for paper: report absolute Cd values as the primary metric;
     use the zero-param baseline as reference (clearly defined geometry), but
     note in a footnote or methods section that the base mesh lies below the
     dataset distribution mean.

  2. Estate bimodal case — regime context.
     The Estate base mesh (Cd=0.223) lies below both distribution peaks (0.31
     and 0.35), meaning it is already in the low-drag aerodynamic regime. The
     optimization is therefore never required to cross a regime boundary — it
     starts in the good basin and exploits it further. This makes the Estate
     problem somewhat easier than it would be if optimizers were initialised
     from the high-drag basin (Cd≈0.35 regime). The tight BO convergence
     (all 20 seeds finding Cd≈0.065) and the bounds-convergence result
     (car_size=0.800, ramp_angle≈+8°) are consistent with this: the problem
     has a well-defined single good basin from the baseline's perspective, even
     though the full design space is bimodal. Regime-crossing is not being
     tested.

  3. Fastback and Notchback — less of a concern.
     F (right-skewed, unimodal) and N (unimodal, narrow IQR) have simpler
     landscape structure. The baseline sitting below the mean is less
     problematic: there is no regime-crossing issue, and the optimizer is
     working in a well-behaved single-basin landscape. Results on F and N are
     therefore easier to interpret cleanly than Estate results.

  4. Should the zero-param baseline still be used as the reference? Yes.
     It is the most defensible reference — a precisely defined geometry
     (all FFD params = 0), not a statistical construct that depends on the
     sampling strategy and LHS bounds. The dataset mean is a useful secondary
     reference for contextualising "improvement over a random design", but the
     zero-param baseline is the correct primary reference for an optimization
     benchmark where the starting point is defined.

METHODOLOGY NOTES (for paper write-up, April 15, 2026)
================================================================================

BENCHMARK SOURCE
----------------
This benchmark uses the DrivAerStar dataset and surrogate framework (Qiu et al.,
NeurIPS 2024). DrivAerStar is a large-scale industrial CFD dataset for vehicle
aerodynamic shape optimization, built on the parametric DrivAer car body. It is
distinct from DrivAerNet++ (Elrefaie et al.), which is a separate dataset also
derived from the DrivAer geometry but using OpenFOAM CFD and a different geometry
parametrization. DrivAerStar uses Star-CCM+ for high-fidelity CFD and provides
VTK surface field outputs (pressure, wall shear stress) alongside a trained
Transolver surrogate model.

The ShapeEvolve benchmark uses all three body variants — Estate (E), Fastback (F),
and Notchback (N) — with the single pre-trained Transolver checkpoint (epoch=490,
1.47M training steps) provided by Qiu et al. The checkpoint was trained on all
three body styles simultaneously (DrivAerStar_3Vehicles dataset, 12,000 training
samples). No fine-tuning or re-training was performed. Drag MAPE on the test set
is 2.64% (E), 2.16% (F), and 2.42% (N).

The primary benchmark (GA/PSO, L-BFGS-B, v3 LLM, BO_torch) was run on the Estate
body only. BO_torch (20 seeds, n=1000) was subsequently run on Fastback and
Notchback as additional test cases (submitted April 15, 2026).

GEOMETRY PARAMETRIZATION
--------------------------
The car surface geometry is deformed using a Free-Form Deformation (FFD) scheme
implemented in Python/NumPy (environments/DrivAer_Star/mesh_generator.py). The
FFD operates on a base VTK surface mesh (the undeformed Estate body) by applying
smooth, region-specific displacements, rotations, and scalings to cell coordinates
via sigmoid-masked interpolation. All 20 parameters are treated as continuous.

The design space is:

  Global shape:
    0.80  <=  s_car         <= 1.20         (uniform scale factor, dimensionless)
    -0.10 <=  Delta_d_i     <= 0.10  m      for i = 1,2,3
                                            (i=1: car width, i=2: car length,
                                             i=3: front bumper length)

  Surface angles:
    -8.0  <=  theta_j       <= 8.0   deg    for j = 1,...,6
                                            (ramp angle, trunk lid angle,
                                             diffuser angle, greenhouse angle,
                                             front hood angle, air intake angle)

  Component position displacements:
    -0.05 <=  delta_k       <= 0.05  m      for k = 1,...,8
                                            (x/z displacements of windscreen,
                                             side mirrors, rear window, trunk lid)

  Tire geometry:
    -0.013 <= Delta_r_tire  <= 0.013 m      (tire diameter offset)
    -0.015 <= Delta_w_tire  <= 0.015 m      (tire width offset)

Total: 20 parameters, all with symmetric box bounds.

FLOW CONDITIONS
---------------
All runs use the following freestream conditions (hardcoded defaults, never
overridden in any submitted job):

    rho_inf  = 1.25  kg/m^3   (freestream density)
    U_inf    = 40.0  m/s      (freestream velocity)
    A_ref    = 2.37  m^2      (reference frontal area)
    q_inf    = 0.5 * rho_inf * U_inf^2  =  1000 Pa   (dynamic pressure)

These correspond to approximately sea-level conditions at 144 km/h.

OBJECTIVE FUNCTION
------------------
The drag coefficient is computed from the predicted surface fields:

    F_drag = sum_i [ p_i * A_i * n_{x,i}  +  tau_{x,i} * A_i ]

    C_D    = F_drag / (q_inf * A_ref)

where p_i is predicted surface pressure, tau_{x,i} is predicted streamwise wall
shear stress, A_i is cell area, and n_{x,i} is the streamwise component of the
outward cell normal. The optimizer reward is r = -C_D (maximized).

Baseline Cd values for the undeformed (all params = 0) bodies:
  Estate (E):   C_D = 0.22334,  C_L = 0.082,  drag = 529 N
  Fastback (F): C_D = 0.18990,  C_L = 0.093,  drag = 450 N
  Notchback (N):C_D = 0.18132,  C_L = 0.110,  drag = 430 N

F and N start ~15% lower than E. The problem is unconstrained (no lift
constraint, no geometric feasibility constraint) — the only constraints are the
box bounds on the 20 design parameters.

SURROGATE MODEL: TRANSOLVER
-----------------------------
The aerodynamic surrogate is Transolver (Qiu et al., NeurIPS 2024), a
transformer-based neural operator for irregular surface meshes. It predicts
per-cell flow fields (pressure and three wall-shear-stress components) from
the surface geometry represented as a point cloud.

  Input:  [N_cells, 7] tensor — cell centroid (x,y,z), cell area, outward
          normal (n_x, n_y, n_z). N_cells ~50,000 for the Estate body.

  Output: [N_cells, 4] tensor — surface pressure p, wall shear stress
          (tau_x, tau_y, tau_z).

  Architecture:
    - Physics-Attention layers: key innovation is clustering cells into learned
      "physics slices" (slice_num=16) that group cells by local flow state rather
      than spatial proximity. Attention is computed within slices, enabling O(N)
      scaling vs O(N^2) for standard attention.
    - 4 transformer layers (n_layers=4), hidden dim 64 (n_hidden=64),
      4 attention heads (n_head=4), GELU activation, no dropout.
    - Input/output normalization from training data statistics. A separate
      norm_stats .pt file is computed per body style from the 10 sample meshes
      in data/vtk_{E,F,N}/. The --norm_stats_path CLI argument selects which
      file to load; defaults to model/norm_stats.pt (Estate).
    - Checkpoint: epoch=490, step=1,473,000 (Lightning checkpoint, extracted to
      transolver_best.pt).

  Inference cost: ~3–10 s/eval on CPU (multi-core), dominated by VTK mesh
  manipulation and forward pass on the ~50k cell point cloud. No GPU was used
  in benchmarking (Transolver falls back to CPU; nodes had no GPU allocation).

  Note: unlike NeuralFoil, Transolver is not differentiable end-to-end from
  design parameters to C_D because the FFD mesh deformation step (mesh_generator.py)
  uses non-differentiable PyVista/NumPy operations. This precludes gradient-based
  post-processing (e.g. adjoint/IPOPT refinement) that was used in the NeuralFoil
  benchmark. All reported C_D values are raw optimizer outputs — no post-processing.

EVALUATION PIPELINE (per oracle call)
--------------------------------------
  1. Design vector (20 floats) -> JSON design file
  2. apply_ffd(base_vtk, params) -> deformed PyVista mesh
  3. extract_model_input(mesh) -> [N_cells, 7] numpy array
  4. Transolver forward pass -> [N_cells, 4] predicted fields
  5. Surface integration -> F_drag, F_lift (scalars)
  6. C_D = F_drag / (q_inf * A_ref); reward = -C_D
  7. Results written to results.csv (one row per eval)

Steps 2–4 dominate wall time. The base VTK mesh is cached in memory across
evaluations (_BASE_MESH_CACHE) to avoid repeated disk I/O; image rendering is
disabled (render_images=False) to eliminate the dominant ~20–60 s/eval overhead
that was present in early testing.

RELATION TO DrivAerNet++
-------------------------
DrivAerNet++ (Elrefaie, Dai, Ahmed; MIT) is a separate but related benchmark
that also uses the DrivAer car body geometry. Key differences:

  Feature             DrivAerStar (this work)   DrivAerNet++
  -------             -----------------------   ------------
  CFD solver          Star-CCM+                 OpenFOAM
  Body styles         Estate, Fast, Notch       Estate, Fast, Notch
  Parametrization     Blender FFD (20 params)   RBF morphing (50 params, approx.)
  Surrogate           Transolver (field)        GNN/PointNet variants
  Dataset size        ~400 samples (training)   ~4000–8000 samples
  Optimization env    Yes (ShapeEvolve)         No (dataset only)
  Differentiable      No (FFD step)             Depends on surrogate

DrivAerNet++ is primarily a dataset and surrogate benchmarking resource; it does
not include an optimization loop. DrivAerStar provides the full pipeline from
parametrized geometry to surrogate inference to reward, making it directly usable
as an optimization environment.

MOTIVATION
  The cd_only benchmark minimizes drag with no aerodynamic constraint.
  We explored adding a lift constraint to create a more physically realistic
  and challenging benchmark. Two candidate formulations were considered:

  (A) Multi-objective: reward = -Cd - β·Cl  (minimize drag + maximize downforce)
      Rejected: requires the surrogate to accurately predict Cl across the full
      design space, including the Cl < -0.10 region that cd_only BO never
      explored. Transolver validation reports MAPE=2.64% for Cd only; Cl
      accuracy outside the training distribution is unknown.

  (B) Constrained drag minimization: reward = -Cd - λ·max(0, Cl - threshold)
      Chosen: the optimizer is still guided by -Cd (the validated quantity).
      Cl only appears in the penalty term, which merely filters out infeasible
      designs. The feasible region (Cl <= -0.10) is well-covered by the 8000-
      sample LHS training data (~13% of uniform random evaluations fall in
      Cl ∈ (-0.20, -0.10) from the cd_only BO runs).

CONSTRAINT BINDING ANALYSIS (from 20k cd_only Estate BO evals)
  Cl <= 0.0  : non-binding — best Cd (0.0651) already has Cl ≈ -0.08. Any λ
               works; constrained and unconstrained optima are identical.
  Cl <= -0.10: binding — best Cd at Cl <= -0.10 is 0.0668 vs unconstrained
               best 0.0651. Constraint forces a real trade-off (~0.002 Cd
               penalty to achieve the downforce requirement).

  λ = 1.0 is sufficient: even λ = 0.1 would enforce the constraint (a design
  at the unconstrained optimum, Cl ≈ -0.08, has violation = 0.02, giving
  penalty = 0.002 > Cd improvement from switching to infeasible region).

REWARD IMPLEMENTATION
  File:      environments/DrivAer_Star/rewards/cd_cl_constrained.py
  Class:     CdClConstrainedReward
  Formula:   reward = -Cd - 1.0 * max(0, Cl - (-0.10))
  Params:    lam=1.0, cl_threshold=-0.10 (both configurable via CLI / config)
  Feedback:  non-empty string with Cl value and penalty when infeasible,
             so the v3 LLM optimizer sees constraint status at each iteration.

CD_ONLY vs. CD_CL_CONSTRAINED: COMPARISON (April 28, 2026)
-----------------------------------------------------------
This section compares the best Cd achieved under two reward variants:
  cd_only:             reward = -Cd  (unconstrained; Cl is free)
  cd_cl_constrained:   reward = -Cd - 1.0 * max(0, Cl - (-0.10))
                        (forces Cl <= -0.10, i.e. net aerodynamic downforce)

The Cl values for cd_only designs are the Cl of the best design found when no
lift constraint is in effect. The optimizer was free to achieve any Cl it wanted;
these are not set by the reward but are the emergent aerodynamic outcome of the
drag-minimizing configuration. Where Cl > -0.10, the design violates the constraint
and would be infeasible under cd_cl_constrained.

cd_only runs used 20 seeds/attempts per method (BO n=1000 evals; v3 n=6000–10000
evals). cd_cl_constrained used 10 seeds/attempts (BO n=1000; v3 n=6000, 54–75%
complete at termination but effectively converged — see plateau check above).

Full comparison table:
  Body  Method          cd_only:           cd_cl_constrained:   Drag
                        best Cd  Cl        best Cd              penalty
  ────  ──────────────  ───────  ────────  ───────────────────  ───────
  E     Bayesian Opt.   0.06509  -0.079*   0.06657              +2.3%
  E     ShapeEvolve     0.06516  -0.065*   0.06970              +7.0%
  F     Bayesian Opt.   0.03064  -0.031*   0.03422              +11.7%
  F     ShapeEvolve     0.03384  -0.094*   0.03492              +3.2%
  N     Bayesian Opt.   0.04956  +0.713*   0.06677              +34.7%
  N     ShapeEvolve     0.06808  -0.118    0.06845              +0.5%

  * = violates Cl <= -0.10 constraint (infeasible under cd_cl_constrained)
  Cl computed from cd_only results.csv: Cl/Cd = Lift/Drag, both in Newtons and
  non-dimensional units respectively (Cl = (lift_N / drag_N) * Cd).

Interpretation:

Estate (E) and Fastback (F):
  The unconstrained optima already have mild negative lift (Cl between -0.03 and
  -0.094), close to but slightly above the -0.10 threshold. Adding the constraint
  forces a modest design change, costing 2–12% in drag depending on method. The
  constraint is binding but not severely: the feasible and infeasible regions are
  adjacent in design space.

Notchback (N) — anomalous BO behavior:
  BO unconstrained finds Cd=0.04956 with Cl=+0.713, a strongly lift-generating
  configuration (the car body is shaped to produce aerodynamic upforce). This is
  a physically valid drag minimizer in an unconstrained setting: the optimizer
  exploits the fact that generating lift reduces induced drag or that a different
  camber distribution reduces pressure drag. However, Cl=+0.713 wildly violates
  the Cl <= -0.10 constraint.

  When BO is constrained (cd_cl_constrained), it must search in the Cl <= -0.10
  region and finds Cd=0.06677 — a 34.7% drag penalty compared to the unconstrained
  optimum. This is not a failure of BO; it reflects the true cost of enforcing
  downforce on the Notchback geometry. The unconstrained and constrained optima
  are in different basins of the design space.

  ShapeEvolve unconstrained converges to Cd=0.06808, Cl=-0.118, which already
  satisfies the constraint. Adding the constraint produces almost no change
  (Cd=0.06845, +0.5%). This is because ShapeEvolve's search trajectory — guided
  by LLM-proposed perturbations — naturally stays in the downforce-generating
  region, whereas BO's acquisition function has no spatial bias and freely explores
  the positive-lift region where drag is lower.

  Implication: for the Notchback body, the two variants (cd_only and
  cd_cl_constrained) are effectively testing different optimization problems. BO
  solves the unconstrained problem better (Cd=0.050 vs 0.068), while ShapeEvolve
  finds a comparable solution to the constrained problem in both cases.

Summary across all bodies:
  BO achieves better best Cd than ShapeEvolve in all constrained cases (best BO
  is 0.4–7.3% lower Cd than best ShapeEvolve). BO also achieves better median
  Cd in 2 of 3 bodies (E: +2.2%, N: +5.7% lower; F comparable +0.7%).
  The performance gap is largest for vtk_E where v3 median (0.079) is notably
  higher than BO median (0.068), suggesting ShapeEvolve has higher run-to-run
  variance under the constrained reward.

  Under the unconstrained cd_only benchmark (where the problems are equivalent
  for E and F), the methods are nearly tied: best Cd differs by < 0.1% for E
  (BO: 0.06509, v3: 0.06516) and 10.4% for F (BO: 0.03064, v3: 0.03384), with
  BO having the edge. For N, the comparison is confounded by BO finding a
  qualitatively different (lift-generating) regime.

DOWNFORCE EFFICIENCY — IMPLEMENTATION AND RUN STATUS (April 29, 2026)
================================================================================

Reward variant: downforce efficiency.

  reward = -Cl / Cd   (i.e. |Cl|/Cd, since Cl < 0 for downforce)

Rationale:
  cd_only and cd_cl_constrained produce nearly identical convergence curves
  because the Cl ≤ -0.10 constraint is weak and rarely active for most designs.
  Downforce efficiency is a fundamentally different landscape: the optimizer must
  simultaneously generate downforce (push Cl negative) and keep drag low — a
  genuine trade-off that should produce visually distinct best designs and
  different convergence behaviour across methods.

  This objective is physically meaningful for performance/GT road cars and avoids
  the degenerate solution of pure downforce maximization (reward = -Cl), which
  can find high-downforce, high-drag shapes that are impractical.

Torchvision fix (April 29, 2026):
  First submission (jobs 8499–8504, April 29) failed immediately with:
    RuntimeError: operator torchvision::nms does not exist
  Root cause: torchvision 0.25.0 in the venv is a CUDA build; its _C.so fails
  to load on CPU-only compute nodes. timm (used by Transolver) imports
  torchvision, which triggers _meta_registrations.py. In torch 2.11.0 the
  @register_fake decorator validates the op exists before registering — but
  with _C unloaded the op is absent, so it raises.
  Fix applied to environments/DrivAer_Star/environment.py: monkey-patch
  torch._library.fake_impl.FakeImplHolder.register to catch "does not exist"
  RuntimeErrors and return a no-op handle, allowing the import to proceed.
  The model (Transolver) does not use any torchvision ops at inference time.
