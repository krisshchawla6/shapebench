DrivAer_Star Benchmark Discussion
April 9, 2026
================================================================================

OVERVIEW
--------
Goal: Benchmark four optimization methods (GA/PSO, L-BFGS-B, BO_torch, v3 LLM)
on a 3D aerodynamic shape optimization problem: minimize drag coefficient (Cd) of
a car body (DrivAer geometry) parameterized by 20 FFD (Free-Form Deformation)
parameters. No constraint on lift — objective is Cd only.

Operating conditions: rho=1.25 kg/m^3, U=40 m/s, area_ref=2.37 m^2.
Reward: reward = -Cd  (maximize reward = minimize drag).

SURROGATE: Transolver — a transformer-based neural network surrogate that predicts
drag and lift from the deformed surface mesh. Input: ~50,000 surface cells with
cell centers, areas, and normals (7 features per cell). Output: Cd, Cl.

Unlike NeuralFoil, Transolver has no analysis_confidence metric and the pipeline
involves a non-differentiable VTK mesh geometry step — so no adjoint/IPOPT
post-processing is possible. Raw optimizer results are the final benchmark outputs.
(See drivaer_star_adjoint_feasibility_April_8_2026.txt for detailed analysis.)

PERFORMANCE OPTIMIZATIONS APPLIED (April 8, 2026)
--------------------------------------------------
Two changes made to DrivAer_Star environment before submitting jobs:

  1. Image rendering disabled (render_images=False default). Eliminated the
     dominant ~20-60 s/eval overhead from off-screen PyVista rendering.

  2. Base VTK mesh cached in memory (_BASE_MESH_CACHE dict). pv.read() called
     once per process; subsequent calls use .copy() on the cached mesh.
     Saves ~1-3 s/eval.

Together: per-eval cost from ~23-70 s → ~3-10 s (CPU). Measured on running
L-BFGS-B jobs: 4.94 s/eval (seed19, 13120 evals over ~18h).


WHY L-BFGS-B IS SLOW
---------------------
The bottleneck is 3-point finite-difference gradients in 20 dimensions:
  - Each gradient step: 2 × 20 = 40 extra forward evaluations
  - At ~5 s/eval: ~3.3 min per gradient, ~3.75 min per L-BFGS-B iteration
  - maxiter=200 × 3 restarts → ~37h total per seed (theoretical max)

Compare to NeuralFoil L-BFGS-B: microseconds per eval → full 3-restart run
in minutes. The eval cost, not the algorithm, is the bottleneck here.

CONVERGENCE ANALYSIS:
18/20 seeds fully converged — all completed all 3 restarts and best Cd flatlined
well before the end of the run (many as early as 25% through). Several seeds
(e.g. 11, 12, 13, 17) found their local minimum and never improved again after
the first quarter of evaluations. The Cd variance across seeds (0.066–0.093)
reflects the distribution of local minima in the 20D landscape, not insufficient
iterations — it is purely initialization-dependent.

2 exceptions:
  seed1:  Δ=0.00327 in last 10% — still improving at termination. Started in an
          extremely poor basin (Cd~0.23) and spent the entire run climbing out
          (0.229 → 0.185 → 0.093). Even with more iterations it is unlikely to
          reach the quality of well-initialized seeds.
  seed3:  Δ=0.00011 in last 10% — marginal, effectively converged.

Running L-BFGS-B longer would not help: the 18 converged seeds have genuinely
found their local optima, and seed1's trajectory is a consequence of poor
initialization that restarts could not overcome.

V3 LLM — NaN PARAMETER BUG (discovered April 12, 2026)
--------------------------------------------------------
During analysis of the n10000 runs, design JSON files were found containing
literal `NaN` values for FFD parameters (e.g. `"side_mirrors_z": NaN`).
NaN is not valid per the JSON spec but Python's json module reads it silently,
so the framework accepted and saved these corrupted files.

Scale of the problem (across 20 n10000 runs):
  side_mirrors_z:       672 NaN occurrences  (anomalously high — ~5× others)
  All other 19 params:  ~117–131 each
  Total corrupted evaluations: ~2,540 / ~200,000 total (~1.3% of budget)

Consequences:
  - Most NaN-param designs → reward=NaN, Cd=NaN in results.csv. The surrogate
    evaluated them, produced invalid geometry, and recorded NaN. Those budget
    slots are wasted.
  - A minority were silently evaluated with NaN treated as 0 by the surrogate
    (e.g. iter_260_o3: NaN in side_mirrors_z but Cd=0.223 recorded). Those
    evaluations ran on a different design than intended.
  - The best designs from v3 are all clean (no NaN) — best Cd is unaffected.
    The n10000 best Cd (0.06516 mid-run) is from a clean design.

Root cause: the LLM uses `NaN` as an implicit "no strong opinion / keep default"
signal for parameters it is uncertain about, instead of a concrete float. The
`side_mirrors_z` parameter is particularly susceptible — the model likely has poor
calibration on its narrow ±0.05 m range relative to the ±8° angular parameters.

Fix: v3 framework should validate all design parameters before calling the
environment (clamp to bounds or resample if NaN/Inf). This is a known failure
mode of structured LLM output: when forced to produce JSON with numeric values,
models occasionally emit NaN rather than a plausible number.

Impact on analysis: NaN-param designs are excluded from all analysis scripts
(convergence curve uses best_reward column which is NaN for failed evals and
thus ignored by the running-max logic; PCA/radar scripts explicitly filter them).

BOUNDS CONVERGENCE (discovered April 12, 2026):
  All four methods independently converged to the same two parameter bounds:
    car_size   = 0.800 (lower bound)   for every best design
    ramp_angle ≈ +8.0° (upper bound)   for every best design
  This indicates both parameters are strongly reward-relevant and that the
  true optimum likely lies outside the current design space bounds. The
  remaining 18 parameters drive the within-bounds variation between methods.
  This is worth revisiting with wider bounds in a future benchmark.

  PCA INTERPRETATION (April 13, 2026):
    The 20-parameter variance is fairly spread: PC1+PC2 together explain only
    29.5%, meaning no single dominant axis captures the design space. This is
    expected — all 20 FFD params have comparable bound ranges.

    Colour-by-Cd (pca_scatter_cd.png) key findings:
      - Low-Cd designs (dark purple in plasma_r = low drag) cluster tightly in
        a specific region, confirming the design space has a well-defined
        low-drag basin. All four method "best" stars land in essentially the
        same region.
      - The low-Cd cluster aligns with the bounds-convergence finding: all
        best designs share car_size=0.800 and ramp_angle≈+8°. PC1 likely
        loads heavily on these two params — it is effectively the "good
        direction" axis.
      - High-Cd designs (yellow) are spread broadly across the rest of the
        space, consistent with drag being sensitive to these key parameters.

    Colour-by-method (pca_scatter_method.png) key findings:
      - v3 (purple): wide, diffuse spread across the entire design space,
        including many high-Cd regions. The LLM explores broadly but
        inefficiently — many evaluations wasted far from the good-Cd basin.
        Despite covering a large area, only 1/20 runs (att13) landed in the
        low-drag cluster.
      - BO_torch (blue): tight, concentrated cluster landing directly in the
        low-Cd region. GP acquisition rapidly narrows the search to the
        promising area. Most evaluations are in or near the good basin — very
        sample-efficient.
      - GA/PSO (orange): broad initial scatter (population initialization
        explores randomly) that converges toward the good region. The spread
        is wider than BO but still reaches the low-Cd cluster for best runs.
      - L-BFGS-B (green): narrow gradient-path traces from multiple starting
        points. Each seed is a thin thread in PCA space — confirms that
        gradient descent takes focused but locally-trapped paths. The starting
        point distribution shows some seeds landing far from the good basin.

    Overall: the PCA confirms that BO_torch's sample efficiency advantage is
    real — it finds and stays in the low-drag basin with ~15–20× fewer evals
    than GA/PSO and L-BFGS-B. The v3 LLM's broad exploration is a double-edged
    sword: it visits more of the space but lacks the acquisition targeting to
    exploit the good region consistently.
