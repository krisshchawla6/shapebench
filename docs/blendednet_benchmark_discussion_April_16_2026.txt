BLENDEDNET BENCHMARK DISCUSSION
Started: April 16, 2026
================================================================================

OVERVIEW
--------
BlendedNet is a Blended Wing Body (BWB) aircraft aerodynamic optimization
environment. It uses a Transolver surrogate (same architecture as DrivAer_Star)
to predict aerodynamic fields, with the shapebench_5 multi-point drag
minimization reward.

This file documents the benchmark runs on BlendedNet across two reward functions:

  shapebench_5 (multi-point CD minimisation):
    Methods: GA/PSO (10 attempts), L-BFGS-B nr3 + nr10 (10 seeds each),
             BO_torch n=500 (10 seeds), CMA-ES n=500 + n500→1000 (10 seeds),
             v3_dynamic_optimizer (10 attempts)
    Status: ALL COMPLETE (April 18–21, 2026)

  shapebench_5_max_LD (multi-point mean L/D maximisation):
    Methods: BO_torch n=500 (10 seeds, COMPLETE), BO_torch n=1000 (10 seeds, ~70% done),
             GA_parallel/PSO 20p×200i (10 seeds, COMPLETE), CMA-ES n=1000 (10 seeds, COMPLETE),
             v3_dynamic_optimizer 200iter×10batch (10 attempts, ~68% done)
    Status: BO n=500, GA_parallel, CMA-ES complete; BO n=1000 and v3 running as of April 22, 2026

REWARD: shapebench_5
--------------------
"shapebench_5" is the 5th benchmark reward configuration defined for BlendedNet
(alongside shapebench_1, ld_ratio.py, etc. in environments/BlendedNet/rewards/).

REWARD EVOLUTION (why shapebench_5 was chosen):
Three reward formulations were developed in sequence before shapebench_5 was
adopted as the benchmark standard:

  1. ld_ratio.py (ld_ratio.py):
     reward = CL_approx / CD_approx  at fixed alpha=5°
     The simplest formulation. Alpha is a free design variable the optimizer can
     exploit; there is no guarantee the design achieves a physically meaningful CL.
     The early GA/particle_swarm exploratory run used this reward (raw L/D ~21–29).

  2. shapebench_1 (sahpebench_1.py):
     reward = -(CD + lambda_cl * |CL_approx - CL_target| + lambda_alpha * alpha_violation)
     Penalized single-point: forces the design toward a target CL and a preferred
     AoA range. Requires manual tuning of lambda_cl and lambda_alpha_range — the
     same lambda-sensitivity problem as the NeuralFoil benchmark's early formulation.
     Used in the shapebecnh1 exploratory runs.

  3. shapebench_5 (shapebench_5.py) ← benchmark standard:
     Multi-point CD minimization via bisection; no penalty weights.
     Bisection solves for alpha that achieves each CL target exactly, then averages
     CD across 5 operating points. Key improvements over shapebench_1:
       - No lambda tuning required: CL targets are met exactly by construction
       - Multi-point: evaluates drag at CL = {0.185, 0.206×3, 0.227}, covering
         a realistic BWB flight envelope rather than a single condition
       - Alpha is not a design variable: it is an output of the bisection, removing
         the optimizer's ability to exploit it
       - Physically motivated: a real BWB must operate efficiently across a range
         of lift coefficients (cruise, climb, loiter); single-point CD ignores this

Objective: minimize mean drag coefficient (CD) across 5 CL operating points.
  CL targets: [0.206, 0.206, 0.206, 0.185, 0.227]
  (3 unique CL values after deduplication: 0.185, 0.206, 0.227)

For each unique CL target, bisection search (8 iterations, alpha in [-5°, 12°])
finds the angle of attack that achieves that CL (using CL ≈ -Cp_mean). This
makes each design evaluation invoke 3 × 8 = 24 Transolver forward passes.

Reward: R = -mean_CD  (mean drag coefficient across the 5 operating points)
Fail reward: -5.0

Note: there are no lift, moment, or geometric feasibility constraints. The
problem is purely drag minimization (unconstrained apart from parameter box bounds).
This contrasts with the NeuralFoil benchmark which has explicit CM and confidence
constraints. Lower mean_CD = higher (less negative) reward = better.

Approximate cost per design evaluation: ~30-50s (24 surrogate calls including
mesh generation on first call + Transolver CPU inference, after mesh cache fix).

--- CMA-ES ---
  CMA-ES (Covariance Matrix Adaptation Evolution Strategy) is the standard
  reference solver for continuous black-box optimization (used as baseline in
  BBOB/COCO benchmarks). It adapts a full covariance matrix over the search
  space, learning parameter correlations and curvature — making it both
  rotation-invariant and scale-invariant. Unlike GA/PSO (axis-aligned) it
  discovers correlated optima (e.g. joint B3/C2 wing loading effects) that
  axis-aligned methods may miss. This makes it the strongest gradient-free
  reference: if v3/GA/BO match CMA-ES, they've found the optimum; if CMA-ES
  beats them, the gap is real and quantifies the cost of ignoring correlations.

  Default population for d=10: λ = 4 + floor(3*ln(10)) = 10 individuals/gen.
  Operates in normalized [0,1] space (sigma0=0.3 is standard for unit-cube).
  n_calls=500 matches BO_torch budget; convergence expected at ~200-400 evals.

  WHY CMA-ES HERE BUT NOT NEURALFOIL OR DRIVAER_STAR:
    NeuralFoil: benchmark COMPLETE; all four methods collapse to the same
      XFOIL L/D (~325-329) after IPOPT post-processing. Optimizer choice
      barely matters — IPOPT dominates. CMA-ES would add no new information.
      Eval cost is negligible (microseconds) but the question is already
      answered.
    DrivAer_Star: BO_torch (20 seeds) already fills the sample-efficient
      reference role. With 20 BO seeds and 20 v3 attempts already run, a
      fifth method adds benchmark bloat without a clear question to answer.
      CMA-ES advantage is also diluted in 20D vs 10D.
    BlendedNet: lbfgsb was broken until April 18; BO_torch just started;
      no established reference for the global optimum exists. CMA-ES on a
      10D smooth surrogate is close to its ideal regime and fills this gap.

  The n=500 convergence plots showed sigma still adapting and the reward trace

  WARM-START APPROACH — design-recovery resume:
    CMA-ES does not persist its covariance matrix to disk, so resuming from a
    checkpoint requires reconstructing the internal state. An ask-based replay
    approach (reseed with same random_state, replay ask()/tell() cycles) was
    attempted first but failed: the cma package version changed between the
    original runs and the resume attempt, causing es.ask() to produce completely
    different proposals (diff norm ~142 in normalised space) despite identical
    seeds. This would corrupt the covariance update.

    Fix: design-recovery approach. For each of the 500 original evals, read
    the stored design JSON file (iter_XXXXX/iter_XXXXX.json), recover the
    normalised vector x_norm = (x_raw - lb) / (ub - lb), and feed those
    directly to es.tell() in popsize=10 batches. This bypasses es.ask() during
    replay entirely, so the covariance update is identical to the original
    regardless of cma version.

    Implementation: frameworks/cmaes/run.py updated with --resume-from flag.
      environments/base.py:             added read_design() abstract method
      environments/BlendedNet/environment.py: implemented read_design()
        → reads CONTINUOUS_KEYS values from design JSON, returns np.ndarray

    Verified (seed 0, n=500, dim=10, popsize=10):
      - All 500 design files loaded without error; all x_norms in [0, 1]
      - sigma after replay: 0.3 → 0.041  (correctly adapted downward)
      - ||CMA mean − best design||: 32 mm  (mean converged near best)
      - es.stop(): {}  (no premature termination; can continue)
      - es.ask() returns valid next proposals after replay

    What these metrics mean:

    sigma (step size) 0.3 → 0.041:
      sigma is the standard deviation of the search distribution in normalised
      [0,1] space. Starting at 0.3 (covering ~30% of each parameter's range),
      a healthy CMA-ES compresses sigma as it zones in on a good region.
      0.041 means the optimizer is now sampling within ~4% of each parameter's
      range — it has learned the scale of variation in the landscape.
      This metric tests whether the covariance update accumulated correctly:
      a corrupted replay (wrong x_norms or rewards) would prevent sigma from
      adapting properly — it would stay near 0.3 (no learning) or collapse
      erratically. The value 0.041 is consistent with typical CMA-ES behaviour
      on smooth 10D problems at n=500 evals (sigma reduction of ~7×).

    ||mean − best_design|| = 32 mm:
      The CMA-ES mean is the weighted centroid of the search distribution,
      updated each generation toward high-reward solutions. After 500 evals it
      should sit near the best region found. The 32 mm distance is measured in
      the raw parameter space (mixed units: mm for B/C, degrees for S/alpha,
      dominated by the large-range B/C parameters with ranges up to 700 mm).
      For context, the diagonal of the full 10D parameter box is ~900 mm; 32 mm
      is ~3.5% of that. This confirms the mean has converged to the correct
      neighbourhood and not drifted to a random region due to corrupted updates.

    Error analysis — two concrete risks checked explicitly:

      Risk 1: clip discrepancy.
        The original run computed x_raw = denorm(clip(x_norm, 0, 1)) but passed
        the unclipped x_norm to es.tell(). The resumed run recovers x_norm from
        the design file and clips before tell(). If any proposal was outside
        [0,1] in the original run, the tell() inputs would differ.
        Checked: x_norm range across all 500 seed-0 evals = [0.000, 1.000].
        No values outside [0,1]. With bounds=[0.0, 1.0] set in cma_options,
        pycma's internal boundary transformation already constrains proposals —
        the clip in the original code was always a no-op. Zero discrepancy.

      Risk 2: JSON float round-trip precision.
        Design parameters are stored as Python float64 in JSON. Checked by
        writing and re-reading the best design (eval 403):
          max |x_raw diff|:  0.00e+00
          max |x_norm diff|: 0.00e+00
        Python's JSON encoder uses enough decimal digits to round-trip float64
        exactly. Zero precision loss.

      Conclusion: the x_norms fed to es.tell() during replay are bit-for-bit
      identical to those in the original run. Since the CMA-ES update equations
      (mean, covariance, evolution paths, sigma) are deterministic functions of
      (solutions, fitnesses), the reconstructed internal state is exact. The
      sigma/mean/stop checks confirm this at the observable level; the error
      analysis above closes the gap at the input level. No unintended errors.

DESIGN OF EXPERIMENTS NOTES
-----------------------------
- L-BFGS-B has the smallest budget (~600 evals) because it uses finite-difference
  gradients: each gradient step internally costs O(dim) = O(10) evaluations.
  The optimizer is informed per step, so fewer raw evals are needed compared to
  population-based methods. Expect fast initial convergence but sensitivity to
  local optima.

- PSO uses 4000 evals, the largest budget. As a population-based global search
  method with no gradient information, it needs more evaluations to explore the
  10D space. The prior 2000-eval run may have been too short to fully converge.
  At ~30-50s/eval (after fix 5), 4000 evals ≈ 33-55h wall time per attempt.

  WHY 20 PARTICLES (not 120 as used in DrivAer/NeuralFoil)
  ---------------------------------------------------------
  Two factors explain the smaller particle count:

  1. Dimensionality. BlendedNet has a 10D design space; DrivAer uses 20 FFD
     parameters and NeuralFoil uses 18 Kulfan coefficients. For 10D, 20
     particles is within the standard PSO recommendation (~10–30 particles for
     low-dimensional problems). 120 particles is more appropriate for 18–20D
     where greater population diversity is needed to cover the larger space.

  2. Evaluation cost. BlendedNet costs ~30–50 s/eval (OpenVSP mesh generation
     + 24 Transolver calls for multi-point CL bisection). Scaling to 120p×500i
     = 60,000 evals would cost ~500–840 h per run — completely infeasible.
     DrivAer and NeuralFoil evaluations are single surrogate or NeuralFoil
     calls in the millisecond range, making 60,000 evals practical for them.

  IS 20 PARTICLES SUFFICIENT?
  ---------------------------
  There is no controlled ablation study comparing 20p vs. more particles at
  the same total budget. The indirect evidence is convergence: observed runs
  flatten by iteration ~150–170 (~3,000–3,400 evals), meaning the final
  600–1,000 evals contribute nothing. This suggests the search is not starved
  for population diversity — it converges before exhausting the budget.
  However, this does not rule out that more particles could reach different
  (better) basins. The 20-particle choice was inherited from the initial
  exploratory run (20p×100i) and never systematically validated.

- v3 uses 2000 evals (200 iter × 10 per iter). The LLM oracle provides
  high-level reasoning about which regions to explore, which can offset the
  smaller raw eval budget. Each iteration: 1 LLM-proposed s0 design + 9
  sequential oracle-guided designs.

- All 3 frameworks are compatible with BlendedNet (no discrete variables,
  get_param_bounds() and write_design() implemented).

- DrivAer_Star BO_torch runs are NOT being run here — BO_torch is already
  running 20 seeds each for vtk_F and vtk_N on DrivAer_Star, and BlendedNet
  does not have a vtk-based parameterization (uses FFD mesh deformation via VSP).
  BO_torch for BlendedNet could be added in a future benchmark pass.

NEW REWARD: shapebench_5_max_LD
--------------------------------

MOTIVATION
The shapebench_5 reward minimises mean drag coefficient (CD) across 3 CL
operating points.  This is a valid aerodynamic objective but it ignores the
lift-to-drag ratio (L/D), which is the fundamental figure of merit for aircraft
range and endurance.  Two designs with identical mean CD but different CL
distributions will score equally under shapebench_5, yet have different real-
world range performance.

The earlier ld_ratio.py reward attempted L/D maximisation but was gameable:
alpha was a free design variable (10-D parameter space includes alpha), so the
optimizer could maximise L/D simply by tuning alpha to the polar peak rather
than improving wing geometry.  The bisection infrastructure introduced in
shapebench_5 eliminates this exploit.

FORMULATION
  max_x  mean_LD(x)

  mean_LD = (1/N) * sum_i  [ CL^(i) / CD(x; CL^(i)) ]
  CL^(i) in {0.206, 0.206, 0.206, 0.185, 0.227}   (same 5-point set as shapebench_5)

  For each unique CL target (0.185, 0.206, 0.227), bisection (8 iterations,
  alpha in [-5°, 12°]) finds the angle of attack that achieves CL^(i) exactly
  (using CL_approx = -Cp_mean from Transolver).  L/D at that trimmed condition
  is then CL^(i) / CD_trimmed.  Mean is taken over the 5 operating points
  (which duplicate CL=0.206 three times, consistent with shapebench_5).

  Reward (maximised by framework):
    R = mean_LD   (dimensionless, expected range ~20–60 for a BWB)

  Fail reward: -5.0 (same as shapebench_5)

  Surrogate calls per design evaluation: 3 unique CL × 8 bisect iterations = 24
  (identical cost to shapebench_5).

RELATIONSHIP TO SHAPEBENCH_5
  shapebench_5:        reward = -mean( CD_i )          — equal weight per operating point
  shapebench_5_max_LD: reward =  mean( CL_i / CD_i )  — each point scaled by its CL target

  These two objectives are NOT equivalent, even though they share identical
  bisection and CL targets.  The distinction is the aggregation:

  shapebench_5 minimises an unweighted mean of CD values.  shapebench_5_max_LD
  maximises a CL-weighted mean of 1/CD values, where the weight for each
  operating point is proportional to its CL target.

  Effective weights from CL_TARGETS = [0.206, 0.206, 0.206, 0.185, 0.227]:
    CL=0.206 → 60%  (appears 3× in the list)
    CL=0.185 → 20%
    CL=0.227 → 20%
  Both rewards share this frequency-based weighting, but shapebench_5_max_LD
  additionally scales each point by its CL value.

  Consequence: a design that trades worse CD at CL=0.185 for better CD at
  CL=0.227 is penalised equally by shapebench_5 (net mean CD unchanged) but
  rewarded by shapebench_5_max_LD (the CL=0.227 point gains more L/D than the
  CL=0.185 point loses).  The two rewards can therefore prefer different
  designs and should not be compared directly.

  Note: a single-point version of each reward (at the same fixed CL) WOULD be
  equivalent, since CL is held constant by bisection and L/D = CL/CD is then a
  monotone function of CD.  The non-equivalence is a multi-point effect arising
  from unequal CL targets.

  Multi-point evaluation is used for the same off-design robustness reasons as
  shapebench_5.

THEORETICAL BASIS AND REFERENCES
  The L/D ratio is the central figure of merit for aircraft aerodynamic
  efficiency, entering directly into the Breguet range equation:

      R = (V/g) × (1/TSFC) × (L/D) × ln(W_init / W_final)

  where V = cruise speed, g = gravitational acceleration, TSFC = thrust-specific
  fuel consumption (weight-based, units 1/s), W_init/W_final = takeoff-to-
  landing weight ratio.  With TSFC, V, and weight fractions fixed as mission
  constants, maximising range reduces exactly to maximising L/D.  For a
  medium-to-long range BWB transport (fuel fraction ~0.35, Liebeck BWB-450
  mission), ln(W_init/W_final) ≈ 0.43.

  Key references:
    Raymer, D.P. — Aircraft Design: A Conceptual Approach, AIAA Education
      Series (multiple editions).  Ch. 3 derives the Breguet equation and uses
      L/D as the primary aerodynamic sizing parameter throughout.
    Anderson, J.D. — Introduction to Flight, McGraw-Hill.  First-principles
      derivation of the Breguet range and endurance equations.
    Torenbeek, E. — Synthesis of Subsonic Airplane Design, Springer (1982).
      Detailed treatment with fuel fraction methods; widely used for transport
      aircraft conceptual design.
    Liebeck, R.H. — "Design of the Blended Wing Body Subsonic Transport",
      Journal of Aircraft, Vol. 41, No. 1, 2004.  Canonical BWB paper; uses
      range and L/D as primary design metrics.  Fuel fraction ~0.35 for the
      BWB-450 long-range mission.
    Qin, N. et al. — "Aerodynamic considerations of blended wing body aircraft",
      Progress in Aerospace Sciences, Vol. 40, 2004.  Surveys BWB aerodynamic
      objectives; L/D and Breguet range are the stated figures of merit.

20 × eval latency). Projected runtime: ~2.3h (200 iters × 41–47 s/iter),

LANDSCAPE VS. CONVERGENCE ANALYSIS — April 22 2026
----------------------------------------------------
Motivation: the cross-metric result (min-CD designs > max-L/D designs on L/D)
raises the question of whether max-L/D runs simply didn't converge, or whether
the L/D landscape has a multiple-basin structure that traps CMA-ES/PSO/ShapeEvolve.

Evidence for landscape trap (not budget/convergence):

1. Convergence plateau confirmed: CMA-ES max-L/D plateaued by e500–700 (of 1000);
   PSO max-L/D plateaued by i60–120 (of 200). Both methods converged well before
   budget exhaustion — additional evals of the same runs would not help.

2. All-seed uniformity: all 10 CMA-ES seeds and all 10 PSO seeds converged to
   the same design corner (Corner B: S1=40°, C2=~675–689, C4=60). If it were
   simply insufficient budget, you would expect variance across seeds. Uniformity
   implies a dominant basin of attraction.

3. BO found Corner A at 500 evals while CMA-ES/PSO failed at 1000/4000 evals:
   BO's acquisition function forces broad exploration; CMA-ES and PSO are
   exploitation-heavy. This asymmetry — BO succeeds, CMA-ES/PSO fail — is the
   classic signature of a hard-to-reach narrow basin, not a budget problem.

Two identified local optima in the max-L/D landscape:
  Corner A (min-CD corner, better L/D):
    S1=60° [MAX], C2=550 [min], C4=90 [MAX], B2~50–65 mm
    Achieved L/D: ~47.8 (BO), ~48.5–48.9 (computed from min-CD best designs)
    Found by: BO max-L/D (directly), all methods via min-CD optimisation

  Corner B (max-L/D corner for CMA-ES/PSO/ShapeEvolve):
    S1=40° [min], C2=~675–689, C4=60 [min], B2~117 mm
    Achieved L/D: 45.7 (PSO), 46.2 (CMA-ES), 46.1 (ShapeEvolve)
    Found by: CMA-ES/PSO/ShapeEvolve when directly optimising L/D

Corner B is the dominant attractor from random initialisation for gradient-free,
exploitation-heavy methods. Corner A requires broad exploration to reach.

Planform interpretation of the two corners:
  Corner A: high inner sweep (S1=60°), narrow fuselage chord (C2=550 min), wide
            tip (C4=90), short mid-section (B2~50–65 mm).
  Corner B: low inner sweep (S1=40°), wider fuselage chord (C2=~680), narrow tip
            (C4=60), longer mid-section (B2~117 mm), wider half-span (~919 vs 851 mm).

The two corners are structurally opposite on three parameters (S1, C4, B2 direction).

LANDSCAPE TRAP FINDING — PAPER DISCUSSION NOTES — April 23 2026
-----------------------------------------------------------------
Bullet points for the shapebench_5_max_LD results discussion in the paper.

THE PHENOMENON
  • shapebench_5_max_LD exposes a landscape trap: the design space contains
    two distinct local optima (Corner A and Corner B) with very different
    basin sizes. All gradient-free methods initialized randomly converge to
    Corner B, while the globally superior Corner A is only found reliably by
    Bayesian Optimization.

  • Corner A geometry: S1=60°[MAX], C2=550[min], C4=90[MAX], B2~50–65mm
    → mean L/D ~48.0–48.9. High sweep, narrow inner chord, maximum tip chord.
  • Corner B geometry: S1=40°[min], C2~675–689, C4=60[min], B2~117mm
    → mean L/D ~45.7–46.2. Low sweep, wider inner chord, minimum tip chord.
  • Corner A achieves ~5–7% higher L/D than Corner B despite all random-start
    methods consistently converging to Corner B.

EVIDENCE FOR THE TRAP
  • All 10 random-start PSO seeds converge to Corner B (best ~45.7).
  • All 10 random-start CMA-ES seeds converge to Corner B (best ~46.2).
  • 8 of 10 random-start ShapeEvolve runs converge to Corner B or worse
    (best ~46.1; median ~37.8 indicating many infeasible evaluations).
  • Only BO (with GP-based exploration) finds designs near Corner A: 2 of 10
    n=1000 seeds reach L/D 47.2 and 47.8. No n=500 seed exceeds 47.0.
  • Warm-start confirmation: PSO, CMA-ES, and v3 all hold at Corner A
    (L/D ~48.0–48.7) when initialized there, confirming it is a stable
    attractor — not a transient or artifact.

WHY CORNER B DOMINATES RANDOM STARTS
  • Corner B has a large basin of attraction: the majority of the normalized
    parameter space rolls toward it under gradient descent / PSO dynamics.
    S1 near its lower bound (40°) and C2 near its middle range are common
    random initializations.
  • Corner A is a narrow shelf: it requires S1 at or near its upper bound
    (60°), C2 at its lower bound (550), and C4 near its upper bound (90) —
    a corner of the parameter space that random sampling reaches with low
    probability (~1/8 of the hypercube corners if all bounds are equally
    likely, and lower in practice due to the basin shape).
  • Once PSO/CMA-ES particles land near Corner B, pbest/gbest updates
    reinforce B, and the optimizer has no mechanism to escape to A.

WHY BO ESCAPES (SOMETIMES)
  • BO's GP acquisition function (expected improvement) maintains a global
    uncertainty model and actively samples unexplored regions, including the
    parameter-space corners where Corner A lives.
  • With n=500, BO does not reliably find Corner A (max ~47.0 across 10
    seeds). With n=1000, 2/10 seeds approach it (47.2, 47.8). This suggests
    Corner A is reachable by BO but requires sufficient exploration budget.
  • BO's advantage here is exploration, not convergence quality: once
    initialized near Corner A (warm-start), PSO and CMA-ES converge just as
    well or better than BO would from the same starting point.

CROSS-OBJECTIVE RELATIONSHIP
  • The same corner (Corner A) is optimal for both objectives:
    - shapebench_5 (min-CD): PSO/CMA-ES/ShapeEvolve all find Corner A reliably
      from random init (it is the dominant attractor for the CD landscape).
    - shapebench_5_max_LD (max-L/D): Corner A is only found by BO or
      warm-started methods.
  • This means the two objectives share the same global optimum but have
    different basins of attraction — the CD landscape guides optimizers toward
    Corner A naturally; the L/D landscape does not.
  • Physically: high sweep (S1=60°) and low inner chord (C2=550) reduce
    induced drag most effectively. The CD objective directly rewards this.
    The L/D objective implicitly rewards it too, but the gradient is weaker
    relative to the large Corner B basin.

BENCHMARK INTERPRETATION
  • shapebench_5_max_LD measures exploration ability more than convergence
    quality. A method that explores broadly (BO) outperforms methods with
    superior local convergence (CMA-ES) purely because it finds the right basin.
  • This is a meaningful property to benchmark: real aerodynamic design spaces
    are multimodal, and the ability to discover narrow high-quality regions
    is practically important.
  • The gap between warm-started and random-start performance (~2.5 L/D units,
    ~5%) quantifies the cost of the landscape trap — i.e., the value of
    exploration in this problem.

DOES shapebench_5 (min-CD) HAVE THE SAME LANDSCAPE TRAP?
----------------------------------------------------------
Short answer: no. The existing data is clear — all random-start methods (PSO,
CMA-ES, ShapeEvolve) already converge to Corner A geometry for min-CD. The
planform plots show essentially the same design found by all methods. There is
no sign of a Corner B trap in the CD landscape, and warm-starting shapebench_5
would provide no benefit.

Physical explanation:

  • min-CD has a cleaner landscape. The dominant drag term at these operating
    points is induced drag, which scales as C_L²/(π e AR). Maximizing span and
    sweep angle (Corner A geometry) reduces this monotonically. There is no
    competing local minimum that trades off span efficiency in a way that
    creates a second basin — the gradient throughout most of the parameter
    space points toward Corner A.

  • max-L/D has a more complex landscape. L/D couples lift and drag
    simultaneously. Corner B (lower sweep, wider inner chord) is a genuine
    local optimum: it generates adequate lift at the prescribed CL* targets
    with moderate drag, forming a real hill in the L/D surface. Optimizers
    get self-reinforced into it — not because it is wrong, but because it is
    locally self-consistent.

  • The two objectives share the same global optimum (Corner A) but not the
    same basins of attraction. The CD landscape is effectively unimodal with a
    dominant Corner A basin. The L/D landscape adds a lift coupling that creates
    Corner B as a distinct, large-basin attractor.

  Conclusion: the landscape trap is specific to the L/D formulation. Warm-start
  investigations are only warranted for shapebench_5_max_LD.

WARM-START CONVERGENCE CURVE INTERPRETATION:

  The warm-start curves do NOT start at the endpoint of the random-start curves.
  This is intentional: the warm-start initial condition is Corner A
  (S1=60°, C2=550mm, C4=90mm, B2≈54mm, L/D≈48.5), which is a completely
  different region of the design space from where random-start methods end up.

  Random-start methods converge to Corner B (~44–46 L/D), a large-basin local
  optimum.  The warm-start experiment bypasses this trap entirely by seeding each
  algorithm directly from the known global-optimum region.  The warm-start initial
  condition is therefore independent of the random-start final state.

  All three warm-start algorithms initially perturb away from the Corner A seed to
  explore, producing offspring that are temporarily worse (e.g. CMA-ES first eval
  ~38 L/D, ShapeEvolve iter_0 offspring ~25–43 L/D).  They then recover and
  converge to ≥48.5 — higher than any random-start method ever reaches.

  For ShapeEvolve (v3) warm-start specifically: the convergence curves are plotted
  from the first new evaluation (iter_0 offspring), excluding the pre-loaded Corner
  A seed design (which lives at iter=-1 in database.json and was not a new function
  evaluation).  This keeps the x-axis consistent with PSO and CMA-ES, which also
  count only new function evaluations from the warm-start condition.

TORCHVISION LOCK FIX — MODULE-LEVEL (April 24 2026)
-----------------------------------------------------

Jobs 3468/3467 (att_2/att_5, the "ext3" resubmission) both failed with the
same torchvision circular import error despite the /tmp lock being in place.

Root cause: the lock was inside _load_model(), which is called lazily on the
first evaluation. But environment.py had already been changed to import
torchvision.extension and Transolver at MODULE LEVEL (lines 24-25). With two
jobs starting simultaneously, both processes hit the module-level import at
the same time, racing on the __pycache__ .pyc write on Lustre before the lock
was ever reached.

Fix: the fcntl.lockf block was moved to wrap the module-level imports:

  with open("/tmp/blendednet_torchvision_import.lock", "a") as _lf:
      fcntl.lockf(_lf, fcntl.LOCK_EX)
      try:
          import torchvision.extension
          from Transolver import Model as _TransolverModel
      finally:
          fcntl.lockf(_lf, fcntl.LOCK_UN)

This serialises the torchvision import at the point where the race actually
occurs. Jobs 4356 (att_2, node 9) and 4357 (att_5, node 8) submitted with
this fix. SLURM also assigned them to separate nodes, providing additional
isolation.
