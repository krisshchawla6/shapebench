BLENDEDNET BENCHMARK DISCUSSION
Started: April 16, 2026
================================================================================

OVERVIEW
--------
BlendedNet is a Blended Wing Body (BWB) aircraft aerodynamic optimization
environment. It uses a Transolver surrogate (same architecture as DrivAer_Star)
to predict aerodynamic fields, with the shapebench_5 multi-point drag
minimization reward.

This file documents the benchmark runs of PSO (GA framework), L-BFGS-B, and
v3_dynamic_optimizer on BlendedNet / shapebench_5.


ENVIRONMENT
-----------
Environment:  BlendedNet
Reward:       shapebench_5
Surrogate:    Transolver (pre-trained, frozen inference)

Design space: 10 continuous parameters
  B1:    [100.0, 200.0]   — fuselage width param 1
  B2:    [50.0,  200.0]   — fuselage width param 2
  B3:    [200.0, 700.0]   — fuselage width param 3
  C2:    [550.0, 850.0]   — wing chord param 2
  C3:    [180.0, 280.0]   — wing chord param 3
  C4:    [60.0,   90.0]   — wing chord param 4
  S1:    [40.0,   60.0]   — surface param 1
  S2:    [40.0,   60.0]   — surface param 2
  S3:    [24.0,   40.0]   — surface param 3
  alpha: [-3.0,    3.0]   — angle of attack (degrees)

No discrete variables.

Flight conditions: Mach=0.3, Re=1e7 (fixed for all runs)


REWARD: shapebench_5
--------------------
Multi-point drag minimization across 5 CL operating points.
  CL targets: [0.206, 0.206, 0.206, 0.185, 0.227]
  (3 unique CL targets after deduplication)

For each unique CL target, bisection search (8 iterations) finds the angle of
attack alpha that achieves that CL. This makes each design evaluation invoke
3 × 8 = 24 Transolver forward passes.

Reward: R = -mean_CD  (mean drag coefficient across the 5 operating points)
Fail reward: -5.0

Approximate cost per design evaluation: ~5-10s (24 surrogate calls at ~0.1-0.5s
each, including bisection overhead).

BENCHMARK RUNS (April 16, 2026)
---------------------------------
Goal: systematic multi-attempt benchmark across 3 frameworks to enable statistical
comparison. Using 10 attempts per framework.

Note on initial failures (April 16, 2026): four successive failure modes were
encountered and fixed before the jobs finally ran correctly:

  1. GA scripts (jobs 2319-2328) passed --random_state which is not a recognised
     run_benchmark.py argument for the GA framework — scripts regenerated without
     it, using attempt-based naming (att1-10).

  2. All 30 jobs (2319-2348 / 2349-2368+2369-2378) failed with:
       FileNotFoundError: ~/miniconda3/envs/openvsp310/bin/python
     BlendedNet mesh_generator.py hardcodes this path via os.path.expanduser.
     The env existed only under /home/ubuntu/miniconda3/envs/openvsp310.
     Fixed by creating a symlink ~/miniconda3/envs/openvsp310 →
     /home/ubuntu/miniconda3/envs/openvsp310 (home is NFS-mounted).

  3. After fix 2, all jobs (2349-2368 / 2382-2401 round) failed with:
       ImportError: libcminpack.so.1: cannot open shared object file
     libcminpack.so.1 is installed on the head node (/usr/lib/x86_64-linux-gnu/)
     but not on the compute nodes. Fixed by replacing the symlink with a wrapper:
       ~/miniconda3/envs/openvsp310/bin/python  (bash wrapper script)
     which sets LD_LIBRARY_PATH to ~/miniconda3/envs/openvsp310/lib/ (containing
     a copy of libcminpack.so.1.3.6) then execs ubuntu's real Python 3.10.
     Verified working on compute node via srun before resubmission.

  4. After fix 3, jobs 2382-2401 ran to completion but almost all evals returned
     FAIL_REWARD (-5.0). Root cause: mesh_generator.py called tempfile.mkstemp()
     for each design but never deleted the STL file after reading. With 20 concurrent
     jobs all writing to /tmp on the same node, /tmp filled up after the first few
     evals per job. SLURM logs confirmed: "[Errno 28] No space left on device:
     '/tmp/tmpXXXXX.stl'". Only eval 0 (before /tmp was full) succeeded per run.
     Fixed by wrapping the mesh read in try/finally and unlinking the temp file:
       environments/BlendedNet/mesh_generator.py  (generate_mesh function)
     This fix applies to all three frameworks (GA, lbfgsb, v3) since all call
     generate_mesh. Re-run with new job IDs (2417-2446).

  Only BlendedNet is affected by fixes 2/3/4 (mesh_generator.py +
  _vsp_generate_stl.py are the only files using the openvsp310 path, and
  the temp file leak is specific to BlendedNet's mesh-based evaluation).

  5. After fix 4, jobs running on dedicated nodes (GA→node-3, v3→node-5) were
     measured at ~4 min/eval instead of the expected ~7s/eval. Root cause:
     environment.py _run_sim() called generate_mesh() on every bisection
     iteration — 24 times per design evaluation (3 unique CL targets × 8 bisect
     steps). The mesh depends only on geometry params (B1–S3); only alpha changes
     between bisection calls. Each generate_mesh() spawns a Python 3.10 openvsp
     subprocess that takes ~4s, so 24 × 4s = 96s of wasted mesh generation per
     eval (plus Transolver inference time → ~4 min total).
     Fixed by adding a process-level in-memory mesh cache in environment.py:
       _mesh_cache: dict  (module-level, keyed by tuple of geometry values)
     _run_sim() checks the cache before calling generate_mesh(); on a cache hit
     the existing pyvista.PolyData object is reused instantly. The cache holds at
     most one distinct mesh at any time (sequential eval) and auto-expires when
     the process exits.
     Expected rate after fix: ~4s (mesh, first call) + 24 × ~1-2s (Transolver
     CPU inference) ≈ 30-50s/eval → ~35-55h for GA (4000 evals),
     ~17-28h for v3 (2000 evals).
     File changed: environments/BlendedNet/environment.py
     GA round 2 (jobs 2447-2456) and v3 round 2 (jobs 2437-2446) cancelled.
     Resubmitted as round 3 with fix 5 applied.

DESIGN OF EXPERIMENTS NOTES
-----------------------------
- L-BFGS-B has the smallest budget (~600 evals) because it uses finite-difference
  gradients: each gradient step internally costs O(dim) = O(10) evaluations.
  The optimizer is informed per step, so fewer raw evals are needed compared to
  population-based methods. Expect fast initial convergence but sensitivity to
  local optima.

- PSO uses 4000 evals, the largest budget. As a population-based global search
  method with no gradient information, it needs more evaluations to explore the
  10D space. The prior 2000-eval run may have been too short to fully converge.
  At ~30-50s/eval (after fix 5), 4000 evals ≈ 33-55h wall time per attempt.

- v3 uses 2000 evals (200 iter × 10 per iter). The LLM oracle provides
  high-level reasoning about which regions to explore, which can offset the
  smaller raw eval budget. Each iteration: 1 LLM-proposed s0 design + 9
  sequential oracle-guided designs.

- All 3 frameworks are compatible with BlendedNet (no discrete variables,
  get_param_bounds() and write_design() implemented).

- DrivAer_Star BO_torch runs are NOT being run here — BO_torch is already
  running 20 seeds each for vtk_F and vtk_N on DrivAer_Star, and BlendedNet
  does not have a vtk-based parameterization (uses FFD mesh deformation via VSP).
  BO_torch for BlendedNet could be added in a future benchmark pass.
