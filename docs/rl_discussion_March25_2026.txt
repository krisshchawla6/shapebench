Reinforcement Learning (RL) Discussion for NeuralFoil
======================================================
Date: March 25, 2026


WHAT RL MEANS
-------------
RL = Reinforcement Learning.

The core idea: instead of the LLM being prompted fresh each iteration with
context, you train an agent (a neural network policy) that learns from
experience which design moves lead to better rewards over time.

In the NeuralFoil context:
  State:   current best design parameters, reward history, constraint violations
  Action:  a perturbation or new set of Kulfan parameters to try
  Reward:  the reward_exact_notebook score
  Agent:   a policy network updated after each evaluation using algorithms
           such as PPO or SAC


HOW IT DIFFERS FROM THE CURRENT LLM APPROACH
---------------------------------------------
                    LLM approach                    RL
                    --------------------            --------------------
Memory              Scratchpad + in-context         Learned policy weights
Adaptation          Via prompt engineering          Via gradient updates
Per-eval cost       LLM API call (expensive)        Forward pass (cheap)
Training cost       None                            Needs many episodes
Interpretability    LLM reasoning is readable       Policy is a black box


ASSESSMENT FOR NEURALFOIL
--------------------------
RL is theoretically appealing but practically hard to justify here because:

1. NeuralFoil evaluations are cheap, but RL needs thousands of episodes
   just to train a useful policy from scratch.

2. 18 continuous dimensions with a noisy reward is a hard RL problem.

3. CMA-ES and Differential Evolution (already on the baseline list) do
   essentially what RL does but more efficiently for this type of
   continuous black-box optimization problem.

Conclusion: RL is more of an interesting research direction than a
practical baseline for the current benchmark. If suggested as a comparison,
it was likely intended as a learned optimizer alternative to the LLM
sampler agent rather than a near-term implementation priority.


FUTURE BENCHMARK FEATURES THAT WOULD MAKE RL WORTH TRYING
----------------------------------------------------------
RL becomes a more compelling option when a benchmark problem has one or
more of the following characteristics:

1. Expensive evaluations (CFD, FEA, wind tunnel)
   If each evaluation takes minutes or hours rather than microseconds,
   the amortized cost of training a policy across many runs becomes
   worthwhile. A trained RL policy can then propose good designs at
   near-zero inference cost, whereas LLM and BO costs scale with every
   call. This is the most important driver — NeuralFoil is too cheap
   for RL's training cost to pay off, but a high-fidelity CFD benchmark
   (e.g. DrivAer, OpenFOAM) would change this calculus.

2. Sequential / multi-step design processes
   If the design task involves a sequence of dependent decisions (e.g.
   designing a wing section by section, or iteratively refining a mesh
   region), RL is a natural fit because it is built around sequential
   decision-making. Single-shot continuous parameter optimization (like
   NeuralFoil's 18 Kulfan parameters) does not exploit this strength.

3. Repeated optimization across a family of problems
   If the benchmark involves solving many related problems (e.g. optimizing
   airfoils across a sweep of Reynolds numbers, Mach numbers, or CL
   targets), a single RL policy can be trained once and transfer across
   the family. LLM and classical optimizers restart from scratch each time.
   This amortization is where RL's training cost becomes justified.

4. Structured or discrete action spaces
   RL handles discrete actions (e.g. topology changes, material selection,
   manufacturing process choices) more naturally than gradient-based or
   Gaussian-perturbation methods. If a future benchmark involves combinatorial
   or mixed discrete-continuous design spaces, RL becomes more competitive.

5. Dynamic or time-varying environments
   If the optimization target changes over time (e.g. optimizing a control
   surface for varying flight conditions in a simulation), RL is the natural
   framework. Static one-shot optimization problems like NeuralFoil do not
   require this capability.

6. Large numbers of runs / ablations planned
   If the research plan involves running hundreds of optimization trials
   (e.g. a large hyperparameter sweep or a multi-seed benchmark study),
   the one-time cost of training an RL policy is amortized across all runs,
   making it increasingly competitive with per-run methods like LLM or BO.

Summary: the single biggest trigger for reconsidering RL is moving to a
benchmark with expensive per-evaluation costs (CFD-scale), where the
training investment pays off across many evaluations or many runs.
