Environment Descriptions: BlendedNet, vlm_3d, fenics_2d
April 8, 2026
==========================================================

OVERVIEW
--------
The shape_evolve repository contains five environments. NeuralFoil and
DrivAer_Star are documented elsewhere. This file covers the remaining three:
BlendedNet, vlm_3d, and fenics_2d.


===========================================================================
1. BlendedNet
===========================================================================

WHAT IT IS
~~~~~~~~~~
Blended-wing-body (BWB) aircraft aerodynamics optimization. The goal is to
minimize drag coefficient across 5 operating lift conditions simultaneously
(multi-point objective, analogous to NeuralFoil's multi-point task). The
reward is the negative mean CD across the 5 CL targets.

DESIGN SPACE
~~~~~~~~~~~~
12 continuous parameters:

    Geometry (9 params):
        B1, B2, B3    — span sections (50-700 mm)
        C2, C3, C4    — chord sections (60-850 mm)
        S1, S2, S3    — sweep sections (40-60 degrees)

    Flight conditions (3 params):
        Re            — Reynolds number
        Mach          — Mach number (default 0.3)
        alpha         — angle of attack (-3 to 3 degrees)

All parameters are continuous with explicit bounds in design_actions.py.

SIMULATOR
~~~~~~~~~
A Transolver neural surrogate (same transformer architecture as DrivAer_Star),
trained on pre-computed CFD solutions. Fully self-contained — weights are
bundled at model/transolver_best.pt. No external solver required.

Pipeline:
    1. Generate mesh from parameters (mesh_generator.py)
    2. Subsample to 8192 mesh points
    3. Transolver inference → per-point Cp, Cfx, Cfz fields
    4. Nearest-neighbor map back to full mesh
    5. Integrate fields to compute drag/lift
    6. Run at each of 5 CL targets via alpha bisection
    7. Return mean CD as objective

Eval speed: ~0.1-0.5s per design on CPU (fast surrogate). Produces Cp/Cfx
visualizations (iso + top view) as output images.

MATURITY
~~~~~~~~
Fully complete. All components are implemented:
    - environment.py: full simulation pipeline and context building
    - design_actions.py: parameter bounds, Gaussian sampling
    - rewards/shapebench_5.py: multi-point objective with alpha bisection
    - prompt_blocks.py: LLM context/response formatting
    - agent.py: LLM action wrappers
    - Transolver model weights: fully trained and bundled

OPTIMIZER COMPATIBILITY
~~~~~~~~~~~~~~~~~~~~~~~
All four NeuralFoil optimizer frameworks work as-is with no modifications:

    PSO (islands/run.py)              YES — ready
    L-BFGS-B (lbfgsb/run.py)         YES — ready
    BO_torch (BO_torch/run.py)        YES — ready
    v3 (v3_dynamic_optimizer/run.py)  YES — ready

Run with: --env BlendedNet (or equivalent environment flag).


===========================================================================
2. vlm_3d
===========================================================================

WHAT IT IS
~~~~~~~~~~
Delta-wing aircraft aerodynamics optimization at transonic/supersonic
conditions. Optimizes the planform and airfoil section of a swept delta wing.
Has both single-point L/D and two-point multi-objective reward options.

DESIGN SPACE
~~~~~~~~~~~~
8 parameters total — mixed continuous and discrete:

    Continuous (5 params):
        le_sweep      — leading-edge sweep angle (45-80 deg)
        root_chord_in — root chord (10-50 in)
        twist_root    — root twist (-10 to 10 deg)
        twist_tip     — tip twist (-10 to 10 deg)
        dihedral      — dihedral angle (-15 to 15 deg)

    Discrete (3 params):
        naca_t        — NACA thickness (integer, 6-24)
        naca_m        — max camber position {0, 2, 4}
        naca_p        — max camber proportion {0, 4}
                        (constrained: only nonzero if naca_m > 0)

    Fixed flight conditions (not optimized):
        aoa           — angle of attack (10 deg default)
        Mach          — Mach number (0.3 default)
        Re            — Reynolds number (3e6 default)

SIMULATOR
~~~~~~~~~
A hybrid physics + learning pipeline:

    1. Vortex Lattice Method (VLM) via the SUAVE library — low-fidelity
       paneled aerodynamic solution
    2. VortexNet GNN — pretrained graph neural network that applies per-panel
       pressure corrections to the VLM solution
    3. VLM re-run with corrected pressure distribution

The GNN weights are in /home/jack/shape_evolve/MF-VortexNet/pretrained_model/
(63 hidden channels, 4 heads, 10 hops). Outputs CL, CDi (induced drag),
CM (pitching moment).

External dependency: SUAVE aerodynamic library must be installed separately.
Eval speed: ~5-10s per design (VLM + GNN, not just a surrogate lookup).

Reward options:
    rewards/single_pt_ld.py    — single-point L/D ratio
    rewards/two_pt_multi.py    — two-point multi-objective with penalties

MATURITY
~~~~~~~~
Fully complete. All components are implemented:
    - environment.py: pipeline integration and context building
    - design_actions.py: Gaussian sampling with discrete parameter snapping
    - Both reward functions implemented
    - prompt_blocks.py and agent.py in place
    - VortexNet model weights fully trained and bundled

OPTIMIZER COMPATIBILITY
~~~~~~~~~~~~~~~~~~~~~~~
The 3 discrete parameters complicate compatibility with classical optimizers:

    PSO (islands/run.py)              PARTIAL — works for the 5 continuous
                                      params; Gaussian sampler handles
                                      discrete snapping correctly. LLM
                                      actions also handle discrete correctly.

    L-BFGS-B (lbfgsb/run.py)         PARTIAL — treats everything as
                                      continuous; discrete NACA params would
                                      not be snapped after each gradient step.
                                      Needs a wrapper to round discrete dims.

    BO_torch (BO_torch/run.py)        PARTIAL — GP-based, designed for
                                      continuous spaces; discrete dims would
                                      need custom acquisition or pre-processing.

    v3 (v3_dynamic_optimizer/run.py)  PARTIAL — LLM optimizer; discrete
                                      handling depends on sampler agent logic.
                                      Likely OK in practice.

To run classical optimizers cleanly, either constrain to the 5 continuous
parameters only, or add a post-step rounding wrapper for the discrete dims.


===========================================================================
3. fenics_2d
===========================================================================

WHAT IT IS
~~~~~~~~~~
2D airfoil shape optimization using a full physics CFD solver. Minimizes
drag with an optional lift penalty on a parametrically deformed airfoil in
incompressible laminar flow (unsteady Navier-Stokes). Adapted from the
Viquerat et al. (2020) deep reinforcement learning shape optimization paper
(cfl-minds/drl_shape_optimization). This is the only environment in the
repository that uses a real PDE solver rather than a neural surrogate.

DESIGN SPACE
~~~~~~~~~~~~
12 continuous parameters:
    - 4 control points on the airfoil, each with 3 deformation values
    - Each value in [-1, 1] (normalized deformation, scaled by max 3.0 units)
    - No named semantics — flat 12D vector
    - Baseline shape is a circular cylinder that deforms toward an airfoil

SIMULATOR
~~~~~~~~~
FEniCS (DOLFIN/mshr) finite-element solver:
    - Unsteady incompressible Navier-Stokes (Taylor-Hood elements:
      P2 velocity, P1 pressure)
    - Domain: rectangular box, default ±15 to ±30 units
    - Mesh: adaptive, generated via mshr from boundary representation
    - Simulation runs to final_time=15s with CFL-adaptive timestep
    - Computes drag and lift via boundary stress tensor integration
    - Outputs: drag, lift, reward (drag penalized by deformation regularity)
    - Visualization: pressure and velocity field VTU files, shape PNG

External dependency: FEniCS (dolfin + mshr) must be installed separately.
This is a non-trivial dependency (typically requires a specific Python/conda
environment or Docker container).

Eval speed: 30 seconds to 2 minutes per design, depending on geometry and
mesh complexity. This is 10-100x more expensive per eval than any surrogate-
based environment in this repository.

MATURITY
~~~~~~~~
The CFD solver and shape pipeline are fully implemented:
    - fenics_solver.py: full unsteady Navier-Stokes PDE solver
    - shapes_utils.py, meshes_utils.py: mesh generation and deformation
    - environment.py: FEniCS integration with singleton lazy-loading pattern
    - design_actions.py: linear and Gaussian sampling, modify actions
    - rewards/default.py: pass-through (solver outputs reward directly)
    - prompt_blocks.py and agent.py: LLM context and action wrappers
    - get_reflection_inputs(): implemented, supporting the v2 reflection cycle

However, get_param_bounds() and write_design() are NOT implemented, which
means the three classical optimizers (PSO, L-BFGS-B, BO_torch) would fail
immediately. Adding these is a ~4-line fix:

    def get_param_bounds(self):
        lb = np.array([-1.0] * 12)
        ub = np.array([ 1.0] * 12)
        return lb, ub

    def write_design(self, x, output_dir, name):
        os.makedirs(output_dir, exist_ok=True)
        path = os.path.join(output_dir, f'{name}.csv')
        np.savetxt(path, [x], delimiter=',', fmt='%.6f')
        return path

The environment appears to have been designed primarily for LLM/v3-style
optimization rather than classical gradient-free methods.

OPTIMIZER COMPATIBILITY
~~~~~~~~~~~~~~~~~~~~~~~
    PSO (islands/run.py)              NO  — get_param_bounds() missing
    L-BFGS-B (lbfgsb/run.py)         NO  — get_param_bounds() missing
    BO_torch (BO_torch/run.py)        NO  — get_param_bounds() missing
    v3 (v3_dynamic_optimizer/run.py)  YES — LLM-only path works as-is;
                                      reflection cycle also supported

All four would work after the ~4-line addition above. Note that at 30s-2min
per eval, BO_torch and L-BFGS-B would be very slow to run to meaningful eval
budgets (L-BFGS-B at 2,500 evals = 20-80 hours on a single CPU).
