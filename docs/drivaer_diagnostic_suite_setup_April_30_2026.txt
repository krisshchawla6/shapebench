PURPOSE
-------
The diagnostic-suite branch provides a ready-to-run LLM-based diagnostic
entrypoint for completed DrivAer_Star runs. Given a single best design, its
flow-field images, and metrics, it constructs a structured prompt and sends
everything to Gemini for analysis. The intended use is to catch:
  - Unusual geometry (drooping rear, bumpy surfaces, bound-saturated params)
  - Missing or implausible constraints
  - Aero anomalies in the pressure/WSS fields
  - Whether the optimizer found a physically meaningful solution

OPEN QUESTIONS / MOTIVATION FOR DIAGNOSTIC
-------------------------------------------
1. 13/20 parameters at their bounds — the optimizer has saturated the design
   space. The diagnostic LLM is expected to flag this and suggest whether
   additional constraints (e.g. minimum Cl, maximum downforce, geometric
   plausibility) should be added to the reward function.

2. Strong downforce (lift = -237 N, Cl ≈ -0.50) despite only minimizing Cd.
   The Cd-only objective with no Cl constraint may be producing aerodynamically
   implausible or impractical designs (a race car posing as a road car).

3. Several geometry angles at extreme values (car_green_house_angle=8,
   car_front_hood_angle=8, car_air_intake_angle=8, diffusor_angle=-8) — the
   LLM should inspect whether the rendered shape looks physically reasonable
   or whether these produce an unrealistic deformation.

4. The diagnostic is expected to suggest constraint additions (e.g. Cl ≥ -0.2
   or |Cl| ≤ some threshold, minimum car_size, plausibility checks on geometry
   angles) that should be added to the cd_only reward for future runs.

FIRST RUN — FIXES APPLIED (April 30, 2026)
================================================================================

Three issues were found and resolved before a clean diagnostic run was obtained.

1. frameworks/.env not present in extracted directory
   The script loads GOOGLE_API_KEY via dotenv from repo_root/frameworks/.env
   (repo_root = /scratch/ShapeEvolve_diagnostic_suite). Since only diagnostic_suite/
   was extracted via git archive, the frameworks/ directory did not exist.
   Fix: symlinked the main repo .env into place:
     mkdir -p /scratch/ShapeEvolve_diagnostic_suite/frameworks
     ln -s /scratch/ShapeEvolve/frameworks/.env \
           /scratch/ShapeEvolve_diagnostic_suite/frameworks/.env
   Note: the .env uses the key name GEMINI_KEY (not GOOGLE_API_KEY); judge.py
   checks all three variants so this works without further changes.

2. LLM response truncated at ~290 chars (finish_reason: MAX_TOKENS)
   diagnostic_suite/config.py had llm_max_output_tokens = 2000. Despite
   the limit, the deprecated google.generativeai SDK with Gemini 2.5 Flash
   was hitting MAX_TOKENS at ~200 tokens of output. Increased to 8192:
     File: /scratch/ShapeEvolve_diagnostic_suite/diagnostic_suite/config.py
     llm_max_output_tokens: int = 8192
   After this fix the LLM returned a complete JSON response (~1800 tokens).

3. base_vtk_path was null — F003/F006 feasibility checks missing evidence
   Neither the design JSON (iter_517_o2.json) nor results.json carries a
   vtk_path field, so model_artifacts["base_vtk_path"] stayed None.
   Fix: added BASE_VTK_PATH to the USER CONFIG block and a fallback override
   after the auto-detection logic in run_drivaer_diagnostics.py:
     BASE_VTK_PATH = "/scratch/ShapeEvolve/environments/DrivAer_Star/data/vtk_E/00000.vtk"
     # after auto-detection block:
     if model_artifacts["base_vtk_path"] is None and BASE_VTK_PATH \
             and os.path.exists(BASE_VTK_PATH):
         model_artifacts["base_vtk_path"] = BASE_VTK_PATH
   With this fix F003 (base VTK exists) and F006 (norm stats compatible with
   body style E) both return status=ok.

Output file structure note:
   The top-level key "feasibility_checks" is not part of the real output schema.
   Feasibility results live at evidence_bundle.feasibility (a list of 6 CheckResult
   objects). A stale empty "feasibility_checks: {}" key appeared in an earlier
   truncated run; it is absent in a complete run. The correct path to inspect:
     diagnostics.json → evidence_bundle → feasibility[0..5]
